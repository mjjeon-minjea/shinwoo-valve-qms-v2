## 2. 기술적 트러블슈팅 (Troubleshooting History)

### [이슈 1] 수동 마이그레이션의 구조적 한계와 휴먼 에러(인적 오류) 리스크
- **원인 (Root Cause)**: 기존 과업에서는 관리자(개발자)가 Supabase 대시보드에서 직접 CSV/JSON을 Export 한 뒤 로컬 서버 폴더로 덮어씌우는 수동 작업을 반복함. 이로 인해 컬럼 누락, 파일 락킹(Locking), 직렬화 포맷 파괴 등 심각한 휴먼 에러 위험성이 상존함.
- **대책 (Solution)**: Node.js의 `fs` 모듈과 Supabase JS Client를 결합하여 `npm run db:pull` 명령어 단 1회 샷으로 7개 핵심 테이블 통신을 일괄(Batch) 처리하는 추상화 엔진 개발. 백업 프로시저(`TIMESTAMP_db_backup.json`)를 선행시켜 덮어쓰기 파괴 방지.
- **결과 (Result)**: 마이그레이션 작업 소요 시간을 1초대로 압축하고, 인적 개입이 완전히 배제된 무결점 데이터 추출 파이프라인 정립 성공.

### [이슈 2] 로컬-클라우드 간 데이터 충돌 (Conflict) 원천 봉쇄
- **원인 (Root Cause)**: 로컬에서 테스트 목적으로 생성된 잉여 레코드(Dummy Data) 모음이 클라우드 실서버로 업로드될 때, 운영 중인 라이브(Live) 데이터베이스와 식별자(PK, ID) 충돌 연쇄를 일으켜 DB 락(Lock)을 유발할 위험성.
- **대책 (Solution)**: `Double-blind Asymmetric Strategy`(비대칭 동기화 전략) 적용. 로컬망의 데이터는 오직 Vercel 실서버의 상태를 1-Way로 수신(`Pull`)하여 복제(Mirroring)하는 데에만 사용하고, 강제 복구(Push) 기능은 물리적으로 제한하거나 단일 데이터 복구 용도의 타겟형으로만 제한 운영.
- **결과 (Result)**: 덤프 충돌 에러가 0건으로 통제되었으며, 개발 과정에서의 안전한 모의 타격이 보장됨.

### [이슈 3] 환경 변수(.env) 보안 격리 및 스크립트 특권 이양 통제
- **원인 (Root Cause)**: 동기화 스크립트는 필연적으로 데이터베이스 전역 열람/편집이 가능한 서비스 롤 키(`SUPABASE_SERVICE_ROLE_KEY`) 또는 고위급 인증 토큰을 필요로 함. 이 권한이 프론트엔드로 노출되면 치명적인 보안 인젝션 대상이 됨.
- **대책 (Solution)**: `sync-db.js`는 클라이언트 환경(Vite 등)과 완전히 격리된 물리적 백그라운드 환경 변수인 `.env.local` 전용 경로만 읽도록 설정. 프론트엔드 환경 변수 프리픽스(`VITE_`)와 백엔드 프리픽스를 분리 파싱.
- **결과 (Result)**: 브라우저 개발자 도구 창으로 어떠한 DB 접근 마스터 키도 유출되지 않음. CLI 터미널 위에서만 보안 프로토콜 유지됨.
