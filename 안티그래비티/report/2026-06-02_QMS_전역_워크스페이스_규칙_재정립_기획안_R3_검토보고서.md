## 4. 세부 분석

### [A] 명시적 승인 키워드 섹션 전역 규칙에 여전히 누락 (높음)

기존 GEMINI.md의 "🚨 명시적 실행 승인 및 AI 행동 통제 제약" 섹션 — 승인/허가/진행/시작/Go 키워드 없이 `write_to_file`, `git push` 등 물리 도구 호출을 원천 차단하는 조항 — 이 R3 전역 규칙 개정안(3-2절)에도 존재하지 않는다.

R1 검토에서 동일하게 지적했으나 반영되지 않았다. 이 조항 없이 전역 규칙이 적용되면 에이전트가 사전 결재 없이 코드 수정을 개시할 수 있는 구조가 된다.

**권고**: 3-2절 전역 규칙에 해당 섹션을 복원하거나, 의도적 삭제라면 계획서에 삭제 사유를 명기할 것.

### [G] release-sync 스킬의 DB 직접 쓰기와 Human-controlled 충돌 여지 (중간)

3-1절 release-sync 스킬 설계안에서 "sqlite3 API 또는 비동기 DB 커넥터를 구동하여 로컬 `shinwoo.db` 내 `release_log` 테이블에 비동기로 직접 INSERT/UPDATE 작업을 수행"한다고 명시했다.

04_harness_constraints.md 2항의 Human-controlled 영역에는 "DB 스키마 직접 수정"이 금지되어 있다. 스키마 변경(DDL)과 데이터 쓰기(DML)는 엄밀히 다르지만, 규칙 문구가 DML까지 포함하는지 불명확하다. 스킬이 자동으로 DB에 INSERT를 수행하면 Human-controlled 원칙과 충돌로 해석될 수 있다.

**권고**: 04 규칙에서 금지 범위를 "DB 스키마 수정(DDL)"으로 명확히 한정하거나, release-sync 스킬의 DB 쓰기가 허용 범위임을 04 규칙 예외 조항에 명시할 것.
