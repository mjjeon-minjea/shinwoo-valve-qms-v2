## 🔍 1. 시스템 현황 진단 및 종합 문제점 분석

현재 QMS v2 "개발자 노트" 시스템을 로컬 서버(db.json)와 실서버(Supabase) 및 프론트엔드 UI 관점에서 면밀하게 진단한 결과, 총 6가지 핵심 문제점이 도출되었습니다.

### 1) 📅 작성일(Date) 포맷 및 백데이트 UI 부재
* **현상**: 로컬 DB에서 신규 승인 시 `date`가 오늘 날짜(`new Date().toLocaleString()`)로 일괄 강제 생성되며, 과거 작업일에 맞춘 릴리즈 이력을 소급(Backdate)하여 보정하는 관리자 UI가 존재하지 않습니다.
* **영향**: 과거 버전(v0.25.1 ~ v2.3.0 등)의 패치노트가 실제 작업일이 아닌 오늘 날짜로 일괄 노출되어 이력 왜곡 및 데이터 신뢰성 훼손이 발생합니다.

### 2) 🚫 반려(Reject) 처리 시 DNAS 검증 편법 우회
* **현상**: `PostApproval.jsx`에서 노트를 반려할 때, JSON Server 미들웨어(DNAS Validator)의 400 에러를 피하기 위해 `content` 필드에 임의로 `(원인: 반려 재작성 / 대책: 본문 보완 / 결과: rejected / 물리적 증빙: 반려 통보)`라는 필수 키워드 텍스트를 인위적으로 강제 주입하고 있습니다.
* **영향**: 밸리데이터 통제 장치를 무력화하는 편법이 소스코드 레벨에 하드코딩되어 있으며, 반려 사유 텍스트가 필요 이상으로 지저분해지고 원본 내용의 무결성이 손상됩니다.

### 3) 🔗 로컬-클라우드 ID 불일치 및 중복 배포 위험
* **현상**: 로컬 DB는 `json-server` 동작에 의해 짧은 문자열(예: `YNyNcBf`)로 ID가 생성되나 Supabase `dev_notes`는 `UUID` 형식을 강제합니다. 현재는 UUID가 아닐 경우 ID를 삭제하고 전송하여 Supabase가 UUID를 자동 생성하도록 우회하고 있습니다.
* **영향**: 로컬 DB의 `id`와 Supabase의 `id`가 상호 불일치하여, 동일 노트를 두 번 배포(Sync)할 경우 중복 적재(Duplicate Row)가 발생하며 일치성 관리가 불가합니다.

### 4) 💾 `is_synced` 배포 상태 필드의 영구 유실 가능성
* **현상**: 로컬 배포 여부를 관리하는 `is_synced` 플래그가 Supabase 테이블 스키마에 존재하지 않습니다. 이 때문에 배포 시 객체에서 해당 필드를 제거(`delete syncData.is_synced`)하여 전송해야 합니다.
* **영향**: 실서버 데이터망 자체에는 배포 상태 플래그가 기록되지 않으므로, 로컬 `db.json` 파일이 삭제되거나 초기화되면 클라우드 배포 여부를 역추적하여 복구할 방법이 전무합니다.

### 5) 🛠️ 윈도우 한글 경로명에 따른 CLI 빌드(Vite) 크래시
* **현상**: `npm run build`를 CMD 및 PowerShell에서 테스트 시 `2843 modules transformed.` 단계 이후 아무런 에러 메시지 없이 빌드가 강제 종료(Exit Code: 1)됩니다.
* **원인**: 워크스페이스 절대 경로에 포함된 한글명(`c:\Users\mjjeon\Desktop\QMS 프로젝트\`)을 Vite/Rollup 및 PostCSS가 인코딩 파싱하는 과정에서 인트라넷 환경 특유의 인코딩 충돌로 무음 크래시가 발생하는 것으로 파악됩니다. (개발 서버 `npm start` 가동에는 문제없으나 배포 빌드 시 제약).

### 6) 👤 담당자 필드 파편화 (`author` vs `manager`)
* **현상**: 데이터 스키마상에서 작성자/담당자를 의미하는 필드가 `author`와 `manager` 두 종류로 혼재되어 사용되고 있습니다.
* **영향**: UI 컴포넌트(`DevNotes.jsx` 및 `PostApproval.jsx`)에서 `item.author || item.manager` 처럼 가변적으로 예외 처리를 해야 하므로 스키마 무결성이 흔들리고 정렬/조회 조건 설정 시 누락 오류를 유발할 수 있습니다.

## 💡 2. 근본적 해결책 제안 (Solution Plan)

각 문제점에 대한 정밀 조치 방안을 제시합니다.

| 분류 | 문제점 | 해결 방안 |
| :--- | :--- | :--- |
| **P1** | **작성일 백데이트** | 로컬 관리자 화면(`PostApproval.jsx`)에 **배포일 수동 입력 인풋** 추가 및 DB 날짜 변경 로직 제공 |
| **P2** | **반려 검증 우회** | 반려 시 `status` 필드를 `rejected`로 업데이트할 때 미들웨어의 DNAS 검증 로직이 `status === 'rejected'`일 경우 검증을 Pass하도록 미들웨어 스크립트(`server.js`) 보완 |
| **P3** | **ID 불일치** | 로컬 패치노트 생성 단계에서 ID를 `crypto.randomUUID()`를 통해 처음부터 UUID 규격으로 생성하도록 통제 |
| **P4** | **배포 상태 유실** | Supabase 테이블에 `is_synced` 또는 배포 상태를 뜻하는 컬럼을 대조하고, 로컬 DB 복구를 위한 클라우드 덤프/풀 싱크 스크립트 작성 |
| **P5** | **빌드 크래시** | 환경 변수 인코딩 설정을 보완하거나 Vite 빌드 캐시 초기화 및 빌드 타겟 경로를 영문 하위 디렉토리로 우회 테스트 |
| **P6** | **담당자 필드** | DB 및 UI 전체에서 담당자 필드를 `author`로 100% 통일하고 `manager` 제거 |
