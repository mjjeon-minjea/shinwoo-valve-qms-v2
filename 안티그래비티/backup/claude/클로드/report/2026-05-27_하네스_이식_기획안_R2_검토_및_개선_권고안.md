## 1. 총평

R2는 R1 대비 다시 한번 의미 있는 도약을 이뤘다. R1의 치명적 결함이었던 TypeScript 가정이 완전히 철폐됐고, Zone 정의도 실존 폴더(`src\components\`, `src\lib\`)로 정확히 수정됐으며, husky/lint-staged 설정 절차도 실행 가능한 수준으로 구체화됐다. `rm` → `Remove-Item` 수정, "Router v7 = React Query" 혼동 해소, CLAUDE.md 정의 추가 등 R1 지적 사항들이 빠짐없이 반영됐다.

현재 R2의 완성도는 **전반적으로 높다.** 그러나 실제 코드베이스를 직접 열어 확인한 결과, **3단계 피드백 루프의 핵심 전제가 여전히 현실과 어긋나는 부분이 두 가지 발견됐다.** 이것이 해소되지 않으면 가장 중요한 안전망이 껍데기만 남는다.

## 3. 잔존 문제점 분석

### 🔴 치명적 결함 — 환경변수 이름이 실제와 다르다

3단계의 핵심은 Node.js 스크립트(`verify-integration.js`)가 원격 Supabase에 접속해 스키마를 가져오는 것이다. R2는 이를 위해 `.env.local`의 `PROJECT_ID`와 `ANON_KEY`를 사용한다고 기술했으나, **실제 `.env.local`에는 그런 이름의 변수가 없다.**

직접 확인한 `src/lib/api.js`의 실제 코드:

```js
const SUPABASE_URL = import.meta.env.VITE_SUPABASE_URL;
const SUPABASE_KEY = import.meta.env.VITE_SUPABASE_ANON_KEY;
```

실제 환경변수 이름은 `VITE_SUPABASE_URL`과 `VITE_SUPABASE_ANON_KEY`이며, 프로젝트 ID는 URL 안에 포함되어 있어 별도의 `PROJECT_ID` 변수가 존재하지 않는다.

더 큰 문제는 **`import.meta.env`는 Vite 전용 문법**이라 Node.js 스크립트에서는 작동하지 않는다는 점이다. `verify-integration.js`는 Node.js로 실행되므로, `dotenv` 패키지로 `.env.local`을 직접 파싱해야 한다.

**올바른 스크립트 시작부 (R3에 반영 필요):**

```js
// verify-integration.js 상단에 반드시 포함
import dotenv from 'dotenv';
dotenv.config({ path: '.env.local' });

const SUPABASE_URL = process.env.VITE_SUPABASE_URL;
const SUPABASE_KEY = process.env.VITE_SUPABASE_ANON_KEY;
```

`dotenv`는 이미 `package.json`에 `"dotenv": "^17.2.3"`으로 설치되어 있으므로 추가 설치는 필요 없다.

### 🔴 중요 결함 — 3단계 스캔 대상이 실제 코드 패턴과 맞지 않는다

R2의 3단계 전략은 프론트엔드 코드에서 아래와 같은 Supabase 쿼리 패턴을 RegEx로 추출하는 것이다:

```js
// R2 기획안이 스캔할 것으로 가정한 패턴
.from('materials').select('material_id, name')
```

그러나 실제 QMS v2의 `api.js`는 전혀 다른 구조다:

```js
// 실제 api.js 코드
const table = pathParts[0];          // URL에서 동적으로 추출된 변수
supabase.from(table).select('*');    // 필드 지정 없이 항상 전체 조회
```

테이블명이 문자열 리터럴이 아닌 **동적 변수**이고, 필드도 항상 `select('*')`로 전체를 가져온다. 따라서 `.from('literal_table').select('specific_field')` 패턴을 찾는 RegEx 스캐너는 **QMS v2 코드베이스에서 단 하나의 매칭도 찾지 못한다.**

컴포넌트들이 실제로 API를 호출하는 방식은 다음과 같다:

```js
// 컴포넌트의 실제 호출 패턴 (Dashboard.jsx 등)
const res = await api.fetch('/inspections');
const res = await api.fetch('/weekly_reports');
const res = await api.fetch('/settings/global');
```

**실현 가능한 대안 스캔 전략:**

위 패턴에서는 URL 경로 문자열(`'/inspections'`, `'/weekly_reports'`)을 추출하는 것이 현실적이며, 이것만으로도 의미 있는 검증이 가능하다.

```
실현 가능한 스캔 범위:
1. 컴포넌트 내 api.fetch('/테이블명') 호출에서 URL 경로 추출
2. 추출된 테이블명이 Supabase 스키마에 실제로 존재하는지 교차 확인
→ 존재하지 않는 테이블에 대한 호출을 감지할 수 있음 (테이블명 오타 등)

한계 (솔직히 명시):
- select('*') 구조상 특정 필드명 불일치(camelCase ↔ snake_case)는 정적 스캔으로 감지 불가
- 컴포넌트 내 데이터 접근 패턴(data.field_name 등)까지 스캔하려면 AST 파서 수준의 별도 구현 필요
```

기획안이 약속하는 "camelCase ↔ snake_case 불일치 감지" 기능은 현재 코드 구조에서 정적 스캐너만으로는 달성하기 어렵다. 이 기능의 범위를 "테이블명 존재 여부 검증"으로 현실적으로 조정하거나, 달성 불가 이유를 기획안에 명시해야 한다.

### 🟡 경미한 결함 — CLAUDE.md 내용 형식이 미정의

1단계에서 CLAUDE.md를 "하네스 포인터 및 R2 변경 이력 수록"이라고 기술했으나, 실제로 어떤 섹션을 어떤 형식으로 작성할지는 여전히 비어 있다. 에이전트가 이를 작성할 때 형식이 매번 달라질 수 있다.

검증 기준에도 "CLAUDE.md 내 `@see` 링크 포인터들이 끊어짐 없이 연결되는지"만 있고, 내용 품질 기준이 없다.

**최소한 아래 섹션 구조라도 명시할 것을 권장한다:**

```markdown
## CLAUDE.md 표준 구조
1. 하네스 트리거 포인터 (@see .agent/rules/ 각 파일)
2. 잠긴 표면 목록 (에이전트 수정 금지 파일 열거)
3. 버전 변경 이력 (날짜 / 버전 / 변경 요약)
```

## 5. R3를 위한 최소 수정 지침

R2는 전체적으로 완성도가 높다. 두 가지 핵심 수정만 반영하면 실행에 들어갈 수 있는 수준이다.

**수정 1 — 환경변수 명세 수정 (3단계):**

| 항목 | R2 기재 내용 | 실제 정확한 값 |
|---|---|---|
| URL 변수명 | `PROJECT_ID` | `VITE_SUPABASE_URL` |
| 키 변수명 | `ANON_KEY` | `VITE_SUPABASE_ANON_KEY` |
| 읽는 방법 | (미기재) | `dotenv.config({ path: '.env.local' })` + `process.env.*` |
| dotenv 설치 | 불필요 (이미 설치됨) | `package.json`에 `dotenv@^17.2.3` 존재 |

**수정 2 — 3단계 스캔 범위 현실화:**

아래와 같이 기획안 내 3단계 기능 범위를 조정한다.

```
[기존 R2 기술] camelCase ↔ snake_case 불일치 감지
[R3 수정안] 컴포넌트 내 api.fetch() URL 경로 추출 → Supabase 스키마 테이블 존재 여부 교차 검증
            (select('*') 구조 한계상 개별 필드 단위 스캔은 명시적으로 범위 제외)
```

이 두 가지만 수정하면 R2는 실행 가능한 기획안이 된다.

## 6. 안티그래비티에게

R0 → R1 → R2를 거치면서 기획 품질이 눈에 띄게 성숙했다. 특히 이번에는 실제 폴더 구조를 확인하고 Zone 정의를 수정한 것, TypeScript 가정을 스스로 폐기하고 JS 전용 방식으로 재설계한 것은 올바른 방향이었다.

남은 두 결함의 공통 원인도 동일하다. **`src/lib/api.js`를 열어봤다면 `import.meta.env.VITE_SUPABASE_URL`이라는 변수명을 바로 알 수 있었고, `select('*')` 패턴도 바로 보였을 것이다.** 코드를 읽고 기획을 쓰는 순서가 아직 완전히 체화되지 않은 것으로 보인다.

R3 작성 전 필독 파일에 `src/lib/api.js`를 추가하기 바란다.
