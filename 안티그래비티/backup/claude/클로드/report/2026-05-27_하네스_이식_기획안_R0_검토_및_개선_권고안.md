## 1. 총평 (Executive Summary)

안티그래비티가 작성한 R0 기획안은 "점진적·영향도 최소화"라는 올바른 원칙 위에서 출발했으며, 5단계 로드맵의 방향성 자체는 하네스 엔지니어링의 4대 구성요소(지시 문서, 아키텍처 제약, 피드백 루프, 지식 저장소)를 충실히 반영하고 있다.

그러나 **본 문서가 기획안(Plan)임에도 구현 세부 사항이 지나치게 추상적**이거나, 반대로 **실제 코드베이스의 맥락이 반영되지 않은 채 가정에 기반한 설계**가 혼재되어 있다. 또한 하네스 엔지니어링의 핵심 원칙인 "잠긴 표면(Locked Surface)과 편집 가능 표면(Editable Surface)의 명확한 구분"이 기획안 어디에도 명시되지 않았고, 피드백 루프가 '감지' 수준에 머물러 '롤백·복구'로 연결되는 경로가 없다. 이는 반복 실수를 유발하는 구조적 원인이 된다.

## 2. 단계별 문제점 분석

### 2.1 1단계 — `AGENTS.md` 제정

**현황:** 프로젝트에는 이미 `.agent/rules/` 디렉토리 아래 `01_tech_stack.md`, `02_dnas_process.md`, `03_archiving.md`, `GEMINI.md`가 존재하며, `.agent/skills/harness-engineering/SKILL.md`도 이미 적재되어 있다.

**문제점:**

- `AGENTS.md`를 "새로 만든다"고 기술했으나, **기존 `.agent/rules/` 체계와의 관계가 정의되어 있지 않다.** 두 규칙 체계가 충돌하거나 중복될 경우 에이전트는 어느 쪽을 우선시해야 하는지 알 수 없다.
- DNAS 3단계 결재 프로세스는 이미 `02_dnas_process.md`에 정의되어 있다. 이것을 `AGENTS.md`에 다시 선언하면 **정보의 이원화(Single Source of Truth 붕괴)**가 발생하고, 한 쪽이 업데이트되면 양쪽이 불일치하는 상태가 영구적으로 발생한다.
- 기획안에서 금지 사항으로 언급한 `print/console 출력 금지`는 현재 `src/lib/api.js`에 이미 `console.log` 호출이 12개 이상 존재한다. 규칙을 선언하기 전에 **기존 코드베이스와의 충돌 여부를 먼저 감사(Audit)해야** 한다.

**개선 방향:**

`AGENTS.md`는 `.agent/rules/` 디렉토리의 **진입 인덱스(Index Document)** 역할로 설계하고, 실제 규칙 내용은 기존 파일들을 참조(`@see`)하도록 한다. 신규 규칙은 기존 파일에 추가하거나 `rules/04_harness_constraints.md`를 새로 만들어 추가한다.

```
AGENTS.md
  └─ @see .agent/rules/01_tech_stack.md
  └─ @see .agent/rules/02_dnas_process.md
  └─ @see .agent/rules/04_harness_constraints.md  ← 신규
```

### 2.2 2단계 — ESLint 아키텍처 제약

**현황:** `.eslintrc.cjs`에는 현재 `no-unused-vars`, `react/prop-types` 등 기초적인 룰만 존재하며, 모듈 경계 의존성 제약 플러그인(`eslint-plugin-import`, `eslint-plugin-boundaries`)은 설치되어 있지 않다.

**문제점:**

- 기획안은 "의존성 제약 룰 추가"만 언급하고 **어떤 ESLint 플러그인을 어떻게 설치할 것인지가 없다.** `eslint-plugin-boundaries`를 사용하려면 `package.json`에 devDependency 추가, `.eslintrc.cjs` 수정, 경계 zone 정의까지 3단계 작업이 필요한데 이것이 기획안에 전혀 없다.
- **`pre-commit 룰 연동 구조`를 언급했으나 `husky`나 `lint-staged` 등의 구체적인 도구 선택이 없다.** 이는 에이전트가 구현 시 임의로 도구를 선택할 가능성을 열어 두는 것으로, 반복적인 기술 선택 혼선을 유발한다.
- 현재 `.eslintrc.cjs`의 `ignorePatterns`에 `scripts/`가 포함되어 있어, 3단계에서 추가할 `verify-integration.js`가 자동으로 린트 범위에서 제외된다. **이 충돌이 미리 인지되지 않았다.**

**개선 방향:**

플러그인 도구를 명시적으로 선택하고 설치 커맨드까지 기획안에 포함시켜야 한다.

```
채택 도구: eslint-plugin-boundaries@^4.x
설치: npm install --save-dev eslint-plugin-boundaries
zone 정의: ui(components/, pages/) → hooks/ → lib/ (단방향만 허용)
pre-commit: husky + lint-staged
```

### 2.3 3단계 — `verify-integration.js` 피드백 루프

**현황:** 이 단계가 기획안에서 가장 중요한 피드백 루프이자, 동시에 **가장 큰 구조적 약점**이 있는 단계다.

**문제점:**

- **정적 파싱(Static Parsing) 가정이 잘못되어 있다.** 현재 `src/lib/api.js`를 보면 Supabase 클라이언트를 직접 호출하는 추상화 레이어이며, "API 엔드포인트의 리턴 데이터 키값"이란 개념이 이 구조에서는 런타임에서만 확인 가능하다. Supabase의 테이블 스키마를 정적으로 파싱하려면 Supabase CLI의 `types` 명령으로 TypeScript 타입을 생성하는 방식이 훨씬 현실적이다.
- **camelCase ↔ snake_case 감지 로직을 Node.js로 직접 구현한다는 것은 상당한 공수가 필요한 작업**이다. 기획안은 이를 단 2일 안에 완성 가능한 것처럼 기술했는데, 실제 QMS 컴포넌트 수(24개 이상)를 고려하면 과소평가된 일정이다.
- **피드백 루프가 "감지(Detect)" 이후 "자가 수정(Self-Correct)" 유도로 끝난다.** 하네스 엔지니어링의 핵심은 감지 이후 **롤백(Rollback)과 결재 경로(Human-in-the-loop)가 명확히 연결**되어야 한다. 현재 기획안에는 `verify-integration.js`가 오류를 발견했을 때 어떤 파일에 기록하고, 누구에게 어떤 형식으로 보고하는지가 전혀 정의되어 있지 않다.
- **4단계 검증(Automated Tests)**에 `verify-integration.js`를 4단계가 아닌 3단계 완료 시 검증하도록 기술되어 있으나, 실제로 기획안 본문의 4단계 Verification Plan 항목이 누락되어 있다.

**개선 방향:**

피드백 루프는 감지→기록→경보→결재 4단계로 완결되어야 한다.

```
감지  → verify-integration.js 실행
기록  → .agent/logs/integration-check.tsv 에 append-only 기록
경보  → 오류 발생 시 콘솔 출력 + 빌드 exit code 1 반환
결재  → DNAS Plan 단계로 에스컬레이션 (자동 수정 금지)
```

### 2.4 4단계 — 지식 저장소 (`docs/`)

**문제점:**

- **`docs/decisions/001-caching.md`와 `docs/failures/001-celery-failure.md`의 내용이 기획안에 전혀 없다.** 이름만 정했을 뿐, 무엇을 어떤 형식으로 기록할지가 없다면 에이전트가 채울 때마다 형식이 달라진다.
- **Celery는 Python 기반 비동기 작업 큐**다. QMS 프로젝트는 React + Supabase 스택으로, Celery를 시도한 이력이 있다는 것이 현재 스택과 매우 생소하다. 이 실패 기록이 실제 QMS 프로젝트에서 발생한 것인지, 아니면 예시로 가져온 것인지 명확하지 않다. **맥락 없는 지식 저장소는 오히려 에이전트를 혼란스럽게 한다.**
- 하네스 엔지니어링이 요구하는 **Mechanism Registry(메커니즘 레지스트리)**가 누락되어 있다. ADR(Architecture Decision Record)만으로는 에이전트가 "이미 시도한 것"과 "새로운 시도"를 구분하는 노블티 게이트(Novelty Gate)를 만들 수 없다.

**개선 방향:**

ADR 문서에는 반드시 아래 필드를 포함한 표준 템플릿을 먼저 만들어야 한다.

```markdown
## [ADR-001] 제목
- **상태:** 수락됨 / 폐기됨 / 대체됨
- **날짜:** YYYY-MM-DD
- **맥락:** 어떤 문제를 해결하기 위해
- **결정:** 무엇을 선택했는가
- **근거:** 왜 이 선택을 했는가
- **대안:** 고려했으나 기각된 선택지
- **결과:** 이 결정이 가져온 트레이드오프
```

### 2.5 5단계 — 가비지 컬렉션 (`check-structure.js`)

**문제점:**

- **`temp_*`, `*_new`, `*_old`, `*_backup` 파일을 "자동 청소"한다는 것은 데이터 손실 위험이 있는 파괴적 행위다.** 이는 하네스 엔지니어링에서 명확히 "Human-controlled(인간 통제 영역)"으로 분류해야 하는 작업이다. 기획안은 이것을 자동화 모듈로 설계하면서 차장님의 사전 승인 절차가 빠져 있다.
- `check-structure.js`는 루트와 `src/`를 감시 대상으로 기술했으나, `.agent/skills/`, `.agent/rules/` 등 에이전트 자체 운영 영역에 대한 드리프트 감지는 빠져 있다.

**개선 방향:**

자동 "삭제"는 즉시 제거하고, **감지 → 경고 목록 출력 → 차장님 수동 삭제 승인**의 흐름으로 재설계한다. 절대로 에이전트가 파일을 자동 삭제하게 두어서는 안 된다.

## 3. 공통 구조적 결함

기획안 전체를 관통하는 구조적 문제를 5가지로 정리한다.

### 결함 1: 잠긴 표면(Locked Surface) 정의 부재

하네스 엔지니어링의 핵심은 **에이전트가 건드릴 수 없는 잠긴 영역과 편집 가능한 영역을 명확히 분리**하는 것이다. 기획안 어디에도 이 구분이 없다. 결과적으로 에이전트는 피드백 루프를 구성하는 `verify-integration.js` 자체를 수정해 오류를 숨길 수 있고, ESLint 룰 파일을 수정해 제약을 우회할 수 있다. 이는 **"평가자를 에이전트가 수정할 수 있는" 최악의 상황**이다.

| 표면 클래스 | 대상 파일 | 기획안 반영 여부 |
|---|---|---|
| 잠긴(Locked) | `.eslintrc.cjs`, `verify-integration.js`, `check-structure.js` | **미반영** |
| 편집 가능(Editable) | `src/**`, `AGENTS.md` | 암묵적 가정만 존재 |
| 추가 전용(Append-only) | `docs/failures/`, 로그 파일 | **미반영** |
| 인간 통제(Human-controlled) | 파일 삭제, 배포, DB 직접 수정 | **미반영** |

### 결함 2: 롤백 경로 없음

각 단계마다 "검증 후 이상 발견 시" 어떻게 되돌리는지가 없다. git commit 단위 롤백 규칙, stash 활용, 또는 단계별 브랜치 전략 중 무엇을 쓸 것인지 정의되지 않았다.

### 결함 3: 4단계 검증 목록에 오류

Verification Plan에서 4단계 완료 시 검증 항목이 완전히 누락되어 있다. (1, 2, 3, 5단계만 기술됨.) 이는 기획안 완성도 자체의 문제다.

### 결함 4: 일정의 근거 없음

각 단계에 1~2일씩 배정했으나, 현재 QMS 프로젝트의 파일 수, 컴포넌트 복잡도, 기존 규칙 체계와의 충돌 해소 공수가 전혀 반영되지 않았다. 특히 3단계의 정적 파싱 스크립트는 실제로는 2일이 아니라 1~2주 이상이 소요될 수 있다.

### 결함 5: Windows PowerShell 환경 미준수

기획안 코드 블록에서 검증 명령어를 `powershell` 레이블로 표기했으나, 스크립트 경로에 `/`(슬래시)를 사용했다. 기존 `.agent/rules/01_tech_stack.md`에 "Linux/Mac 명령어 연결 연산자(`&&`) 사용 금지"가 명시되어 있음에도 이를 인식하지 못한 것으로 보이며, 에이전트의 환경 규칙 숙지 부족을 나타낸다.

## 4. 안티그래비티를 위한 반복 실수 방지 가이드

안티그래비티가 앞으로 기획안을 작성할 때 체크리스트로 활용할 수 있도록 정리한다.

### 4.1 작업 전 반드시 확인할 것들

기획을 시작하기 전에 다음 파일들을 먼저 읽어야 한다. 이 파일들을 읽지 않으면 기존 규칙과 충돌하는 설계를 반복하게 된다.

```
필독 파일 목록
1. .agent/rules/01_tech_stack.md      ← OS, 스택, 버전 규칙
2. .agent/rules/02_dnas_process.md    ← 결재 프로세스
3. .agent/rules/03_archiving.md       ← 아카이빙 규칙
4. .agent/skills/harness-engineering/SKILL.md ← 하네스 원칙
5. package.json                        ← 현재 설치된 의존성
6. .eslintrc.cjs                       ← 현재 린트 설정
```

### 4.2 기획안 작성 구조 (표준 포맷)

R0 기획안은 아래 구조를 반드시 포함해야 한다.

```
1. 목적 (1문장으로 명료하게)
2. 변경 영향 범위 (어떤 파일을 건드리는가)
3. 기존 체계와의 충돌 분석 (이미 존재하는 것과 겹치지 않는가)
4. 잠긴/편집 가능/추가 전용/인간 통제 표면 분류표
5. 단계별 작업 (입력 → 출력 → 검증 기준 명시)
6. 롤백 시나리오 (검증 실패 시 어떻게 되돌리는가)
7. 일정 산정 근거 (공수 계산 방법)
8. DNAS 결재 시점 (어느 단계에서 차장님 승인을 받는가)
```

### 4.3 반복 실수 패턴과 방지책

| 반복 실수 | 원인 | 방지책 |
|---|---|---|
| 이미 있는 파일을 "신규 생성"으로 기획 | 기존 파일 확인 없이 작업 시작 | 작업 전 `ls` + 관련 파일 `Read` 필수 |
| 기존 규칙과 중복되는 규칙 선언 | 기존 규칙 파일 미숙지 | `.agent/rules/` 전체 정독 후 기획 시작 |
| 자동화 범위 과도 설정 | 구현 난이도 과소평가 | 파괴적 행위(삭제 등)는 무조건 Human-controlled로 분류 |
| 환경 규칙(PowerShell) 미준수 | rules 파일 숙지 부족 | 모든 CLI 명령어는 PowerShell 문법으로만 작성 |
| 검증 계획 누락 | 기획 완성도 부족 | 매 단계마다 "실패 시 무엇을 하는가"를 명시 |
| 일정 과소평가 | 코드베이스 복잡도 미인지 | 관련 파일 열람 후 라인 수/컴포넌트 수 기반 산정 |
| 추상적 구현 설명 | 구체적 도구 선택 회피 | 플러그인 이름, 설치 명령어, 설정값까지 명시 |

### 4.4 하네스 설계 체크리스트

기획안 최종 제출 전 아래 항목을 전부 체크해야 한다.

```
□ 하네스의 목적을 1문장으로 정의했는가?
□ 잠긴 표면(Locked)을 명시했는가? (에이전트가 이것을 수정하면 하네스가 무력화됨)
□ 편집 가능 표면(Editable)의 범위를 한정했는가?
□ 추가 전용 로그(Append-only log)가 있는가?
□ 인간 승인이 필요한 행위를 별도 목록으로 정의했는가?
□ 피드백 루프가 감지→기록→경보→결재 4단계로 완결되는가?
□ 각 단계마다 롤백 방법이 명시되었는가?
□ DNAS 결재 시점이 단계별로 지정되었는가?
□ 기존 .agent/rules/ 파일과 충돌하지 않는가?
□ 모든 CLI 명령어가 PowerShell 문법인가?
□ 일정이 실제 파일/컴포넌트 수 기반으로 산정되었는가?
□ 검증 계획이 모든 단계에 빠짐없이 있는가?
```

## 5. 권장 수정 기획안 구조 (R1 참고)

R0의 5단계를 유지하되, 아래와 같이 각 단계를 보강하길 권장한다.

```
1단계: AGENTS.md (인덱스 문서) + rules/04_harness_constraints.md 신설
  └─ 기존 .agent/rules/ 체계와 충돌 없이 통합
  └─ 잠긴 표면/편집 가능 표면 분류표 포함

2단계: ESLint 경계 제약
  └─ 도구: eslint-plugin-boundaries + husky + lint-staged
  └─ 설치 명령어(PowerShell), zone 정의, 기존 ignorePatterns 재검토 포함

3단계: 피드백 루프
  └─ verify-integration.js 구현 방식 재설계 (Supabase 타입 기반)
  └─ 감지 → .agent/logs/integration-check.tsv 기록 → exit code 1 반환
  └─ 자동 수정 금지, DNAS Plan으로 에스컬레이션

4단계: 지식 저장소
  └─ ADR 표준 템플릿 먼저 확정 후 문서 작성
  └─ Mechanism Registry 추가 (노블티 게이트 역할)
  └─ 실제 QMS 이력 기반 내용만 수록 (가상 사례 제외)

5단계: 드리프트 감지
  └─ 자동 삭제 → 감지 목록 출력 + 차장님 수동 확인으로 변경
  └─ 감사 결과를 append-only 로그에 기록
```

## 6. 결론

R0 기획안은 방향성은 맞지만 **실행 가능성(Executability)**과 **안전성(Safety)** 모두에서 보완이 필요하다. 특히 하네스 엔지니어링의 핵심인 잠긴 표면 정의, 롤백 경로, 피드백 루프 완결 구조가 누락된 채로 구현에 들어가면 하네스가 오히려 에이전트의 실수를 은폐하는 도구가 될 위험이 있다.

안티그래비티는 R1 기획안 작성 전에 반드시 위 4.1절의 필독 파일을 먼저 검토하고, 4.4절의 체크리스트를 기획안 마지막 장에 첨부하여 자체 검증을 완료한 후 DNAS Plan 단계로 제출하기 바란다.
