## 🔄 피드백 루프 안전장치 (Feedback Loop)

Gemini API 및 원격 시트 통신의 오류 가능성에 대비해 철저한 **4단계 피드백 루프**를 설계합니다:
1. **감지 (Detection):** 구글 시트 로드 실패, Gemini API 스로틀링(Throttling), Supabase 트랜잭션 에러 발생 시 즉각 예외를 Catch합니다.
2. **기록 (Logging):** 에러의 원인, 발생 시점, 대상을 DB 내 `system_logs` 테이블 또는 로컬 콘솔 로그에 정밀하게 기록합니다.
3. **경보 (Alert):** 동기화 오류 발생 시 QMS 메인 대시보드 및 인수검사 화면 상단에 **[⚠️ 구글 동기화 일시 오류 발생 (이전 저장된 캐시 데이터 표시 중)]** 경보 배너를 미려하게 노출하여 차장님께 즉각 공유합니다.
4. **결재 (Approval/Manual Sync):** 오류 상태에서 차장님께서 대시보드의 **[수동 즉시 동기화 및 재시도]** 버튼을 통해 재연동 명령을 인가할 수 있는 결재형 컨트롤 인터페이스를 제공합니다.

## 🛡️ 3단계 배포 파이프라인 및 단계별 롤백 방안 (Rollback Plan)

```
[1단계: 로컬 포트 가상 시뮬레이션] ➔ [2단계: 테스트웹(Staging DB) 배포] ➔ [3단계: 메인웹 최종 적용]
```

### 1단계: 로컬 가상 검증
* **검증:** `npm run dev` 로컬 Vite 서버 상에서 수동 API 호출을 모의 구현(Mocking)하여 UI 그리드 및 조치율 분석 차트의 색감/정렬 1차 점검.
* **롤백 방안:** 로컬 코드 변경 사항이 비정상일 경우, Git 로컬 체크아웃(`git checkout src/`)을 수행하여 1초 만에 즉시 원복합니다.

### 2단계: 테스트웹 배포 검증
* **검증:** GitHub 테스트 저장소(`qms.git`) ➔ 테스트 Vercel(`shinwoo-valve-qms.vercel.app`) ➔ Supabase 스테이징 DB(`srzaanvojyhwzugoaimk`) 연동 후 실제 구글 시트 원격 동기화 작동 여부 2차 크로스 검수.
* **롤백 방안:** 스테이징 서버 동작 비정상 시, Vercel 어드민 콘솔에서 **"이전 성공 빌드 버전(Previous Deployment)"으로 즉시 원클릭 롤백** 처리합니다.

### 3단계: 메인웹 최종 배포
* **검증:** 차장님의 2단계 승인 획득 후, GitHub 실서버 저장소(`qms-v2.git`) ➔ 메인 Vercel(`shinwoo-valve-qms-v2.vercel.app`) ➔ 실서버 Supabase DB 배포 및 전체 운영계 사이트 최종 개방.
* **롤백 방안:** 실서버 장애 감지 시, 메인 Vercel 즉각적 이전 빌드 롤백 실행 및 Supabase 마이그레이션 백업 파일(`db_backup.sql`)을 기동하여 복구 작업을 원자적으로 수행합니다. (파괴적 행위 발생 시 차장님 승인 필수 적용)
