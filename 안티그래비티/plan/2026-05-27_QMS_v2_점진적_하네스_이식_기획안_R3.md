### 3단계: [피드백 루프] `verify-integration.js` 테이블 검증 센서 (JS 전용)
* **목표:** `select('*')` 동적 변수 패턴과 Vite 전용 환경변수를 고려하여, 런타임에 실패하지 않는 원격 Supabase 테이블 스키마 존재성 검증을 현실화합니다.
* **구현 사양:**
  - **Node.js 환경변수 읽기 파싱:** Vite 전용 문법(`import.meta.env`)을 전면 폐기하고, package.json에 기포함된 `dotenv`를 활용해 백엔드 런타임 환경에서 안전하게 `.env.local` 변수를 로드합니다.
    ```js
    import dotenv from 'dotenv';
    dotenv.config({ path: '.env.local' });
    
    const SUPABASE_URL = process.env.VITE_SUPABASE_URL;
    const SUPABASE_KEY = process.env.VITE_SUPABASE_ANON_KEY;
    ```
  - **원격 Supabase 메타데이터 동적 수집:** 수집한 원격 URL과 Key를 기반으로 Supabase REST API(`https://<PROJECT_ID>.supabase.co/rest/v1/`) 스키마 엔드포인트에 `node-fetch`로 HTTP 요청을 보내어 데이터베이스 내 **실제 테이블명 목록**을 수집하여 JSON 메타데이터(`_workspace\supabase_schema.json`)로 보존합니다.
  - **프론트엔드 URL 호출 스캔:** `src\components\` 내의 모든 UI 컴포넌트들을 Node.js 정적 스캐너로 스캔하여, 컴포넌트들이 `api.fetch('...')` 형태로 호출하는 **경로 내의 테이블명 목록**을 RegEx를 활용하여 추출합니다.
    - *추출 정규식:* `api\.fetch\(\s*['"]\/([^'"]+?)['"]`
  - **테이블 존재성 정합성 크로스 비교:** 추출된 호출 테이블명들이 원격 Supabase 실제 테이블명 목록에 존재하지 않거나 오탈자(예: `/inspectionss`)가 있을 경우 즉시 에러를 쏩니다.
  - **기술적 한계 솔직 명시:** `src\lib\api.js` 내부가 항상 `select('*')` 형태로 테이블 전체를 조회하고 문자열이 아닌 동적 변수로 데이터베이스를 조작하므로, **정적 스캔을 통한 개별 필드명 불일치(camelCase ↔ snake_case) 검증은 기술적으로 불가능함**을 기획서에 투명하게 명시하여 불필요한 공수와 허위 규칙 선언을 영구 방지합니다.
  - **피드백 완결 루프 (4단계 완결):**
    1. **감지:** `verify-integration.js` 스크립트를 구동하여 존재하지 않는 테이블 호출 스캔.
    2. **기록:** 검사 결과를 `.agent\logs\integration-check.tsv` 파일에 append-only 이력으로 기록.
    3. **경보:** 불일치 감지 즉시 빌드 exit code `1`을 방출하며 콘솔에 오류 상세 위치(파일:라인) 표시.
    4. **결재 에스컬레이션:** 에이전트의 자동 수정을 금지하고 즉시 차장님 수동 확인을 위한 DNAS Plan 단계로 보고 에스컬레이션.
* **공수 산정 근거:** 동적 REST API 스캔 정규식 작성 및 `node-fetch` 연동 스키마 검증기 구현과 QMS v2 24개 컴포넌트 드라이런에 총 5일의 공수를 현실적으로 배정합니다.
* **검증 기준:** 의도적으로 잘못된 테이블 호출 코드(`src\components\test_mismatch.js` 내 `api.fetch('/invalid_table')`)를 심고 스크립트를 실행하여 exit code `1`이 성공적으로 터져 나오는지 검증.
