# QMS v2 배포 인프라 정비 구현계획서 (R2)

**Goal:** 로컬-테스트웹-메인웹 3계층 배포 인프라를 정비하고, 원격 브랜치 분할 추적 설정 및 단방향 데이터 복제 자동화 스크립트를 구현하여 안전한 릴리즈 체계를 수립한다.

**Architecture:** 로컬 개발은 테스트 Supabase 환경변수를 활용하며, 테스트웹은 origin/staging 브랜치를 추적하고 메인웹은 origin/main 브랜치를 추적하도록 배선한다. 데이터는 메인 Supabase에서 Staging Supabase로만 흐르는 단방향 동기화(Read-Only 복제)를 적용한다.

**Tech Stack:** React(Vite SPA), Supabase, Vercel, Git/GitHub, PowerShell

#### 메인 DB 쓰기 물리 차단 메커니즘 (반려 피드백 [D] 핵심)

`clone-db.js`에서 메인 DB(소스)로의 쓰기를 원천 봉쇄하는 **2중 방어 코드**를 적용합니다:

**방어 1: Supabase JS 클라이언트 Proxy 래핑 — CUD 메서드 호출 시 즉시 예외**
```javascript
// 메인 DB 읽기전용 보호 래퍼
function createReadOnlyClient(url, key) {
  const client = createClient(url, key, { auth: { persistSession: false } });
  
  // from() 반환 객체에서 insert/update/upsert/delete 호출 시 throw
  const originalFrom = client.from.bind(client);
  client.from = (table) => {
    const queryBuilder = originalFrom(table);
    const BLOCKED = ['insert', 'update', 'upsert', 'delete'];
    for (const method of BLOCKED) {
      queryBuilder[method] = () => {
        throw new Error(`🚫 [BLOCKED] 메인 DB(prod)에 대한 ${method}() 호출이 물리 차단되었습니다. table=${table}`);
      };
    }
    return queryBuilder;
  };
  
  return client;
}
```

**방어 2: 스크립트 로직 구조 — prodDb 객체는 select()만 호출**
```javascript
// prodDb: select('*')만 호출 (읽기 전용)
const { data, error } = await prodDb.from(table).select('*');

// stagingDb: upsert만 호출 (쓰기 대상)
const { error: upsertErr } = await stagingDb.from(table).upsert(data);
```

**소스 DB 지정 원칙:** `clone-db.js`의 소스(prod) DB 변수는 `PROD_DB_URL`이 아닌, **검증된 메인 DB Supabase HTTP URL**을 사용합니다. 단, 이 값은 **Task 1(W1) Phase 3에서 차장님이 최종 확정한 메인 DB**가 결정된 이후에만 설정합니다. W1 미확정 시 clone-db.js 실행은 차단됩니다.
