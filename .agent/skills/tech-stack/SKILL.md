# ⚙️ 기술 스택 및 코딩 표준 규칙 (tech-stack)

본 규칙은 신우밸브 QMS v2 개발 환경에서 사용하는 기술 스택 및 코딩 행위를 제약하는 강제 규정입니다.

## 1. 기술 스택 및 아키텍처 규격
- **프론트엔드/백엔드**: React(Vite/SPA), 백엔드 및 데이터베이스는 Supabase만 사용합니다.
- **제한사항**: MySQL, MongoDB, Node.js(Express) 등의 타 프레임워크나 DB 도입 제안을 금지합니다.
- **3단계 격리 원칙**: [Local(개발) -> Staging(테스트) -> Production(운영)] 순서의 3단계 물리 격리 배포 원칙을 고수합니다.

## 2. OS 및 터미널 명령어 제약 (Windows PowerShell)
- **명령 결합 연산자 제한**: CLI 명령어 설명 시 리눅스/맥용 명령어 결합 연산자(`&&`)를 사용해서는 안 되며, 오직 PowerShell 문법만 허용합니다.
- **삭제 명령어 제한**: 파일이나 폴더의 삭제는 리눅스용 `rm`이 전면 금지하며, 반드시 PowerShell 전용 명령어인 `Remove-Item`을 사용해야 합니다.

## 3. 코드 작성 및 에러 대응 규칙
- **코드 경로 명시**: 코드 수정안을 제시할 때는 반드시 해당 소스 파일의 정확한 경로를 코드 블록 상단 주석에 기재합니다.
- **코드 생략 금지**: 코드베이스 복붙 시 '여기서부터 알아서 구현' 식의 생략(...)을 금지하며, 완제 코드를 제공합니다.
- **에러 추측 금지**: 에러 원인을 추측하지 않고, 브라우저 콘솔(F12) 로그 및 네트워크 탭의 응답 코드를 전민재 차장님께 요청하여 팩트 기반으로 분석합니다.
- **환경 변수 통제**: Node.js 스크립트(서버/DB 등) 환경에서는 프론트용 `import.meta.env` 사용을 전면 금지하며, 반드시 `dotenv` 라이브러리를 로드하여 `process.env`로 제어합니다.
- **Supabase 변수명 고정**: Supabase 관련 환경 변수 정의 시 `PROJECT_ID`나 `ANON_KEY` 같은 자의적 이름을 금지하며, 반드시 시스템 표준명인 `VITE_SUPABASE_URL`, `VITE_SUPABASE_ANON_KEY`로 명명합니다.

## 4. [조항 4] AI 코딩 행동 고삐 규격
- **Simplicity First (과설계 방지)**: 요청 사항에 없는 추상 클래스, 장황한 인터페이스 설계, 디자인 패턴 남용, "나중에 쓰일 기능"은 단 한 줄도 코드에 적지 않습니다. 50줄로 해결될 코드를 200줄로 늘리지 마십시오.
- **Surgical Changes (수술식 최소 변경)**: 소스코드를 수정할 때, 지정된 요건 범위 외의 인접한 멀쩡한 코드의 인덴팅, 줄바꿈, Adjacent Comments를 임의로 재정리하지 마십시오. 원본 파일 스타일을 100% 모방합니다.
- **Ollama IP 127.0.0.1 강제 규칙**: 로컬 AI API 연동(11434 포트) 시에는 IPv6 교착 에러를 막기 위해 `localhost` 사용을 금지하고 반드시 `127.0.0.1`을 강제하되, 일반 로컬 Vite 개발 서버 및 API는 기존의 환경 변수(Vite 기본값)를 철저히 이격 보존하여 침범하지 않습니다.
