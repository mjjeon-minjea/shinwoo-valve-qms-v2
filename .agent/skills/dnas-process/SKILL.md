# 📋 DNAS 결재 및 배포 프로세스 규칙 (dnas-process)

본 규칙은 신우밸브 QMS v2 개발에서 발생하는 의사결정 기록, 승인 획득, 그리고 물리적 배포 행위를 통제하는 강제 규정입니다.

## 1. 3단계 결재 프로세스 (DNAS)
에이전트는 기능 추가나 수정 작업을 진행할 때, 다음의 3단계 문서 상신 및 승인 획득 절차를 절대 생략하거나 건너뛸 수 없습니다.

1. **`Plan` (구현 계획서 상신)**:
   - 변경 대상 파일, 시스템 충돌 위험성을 분석한 구현 계획서(`implementation_plan`) 문서를 작성하여 차장님의 승인 결재를 받아야 합니다.
   - 송출 직전 `plan-self-review` 스킬을 실행하여 높음(High) 등급 이슈가 0건인지 검증하고, 확정된 Plan은 `revision-archiver` 스킬을 사용하여 `안티그래비티\plan\` 폴더에 리비전 파일로 아카이빙해야 합니다.
   - 구현 계획서는 방향성이 완전히 합의되고 차장님께서 명시적으로 **"플랜 가져와라"**고 오더하셨을 때에만 정식으로 수립합니다. (사전 독단 작성 금지)

2. **`Task` (작업 명세서 상신)**:
   - 기획안이 승인(결재 완료)되면 실제 코드를 수정하기 전, `task` 규격의 세부 작업 체크리스트 문서를 수립하여 상신합니다.
   - 확정된 Task는 `revision-archiver` 스킬을 사용하여 `안티그래비티\task\` 폴더에 리비전 파일로 아카이빙합니다.

3. **`Walkthrough` (최종 완료 보고서)**:
   - 개발 및 시운전 검증이 완료되면 최종 결과를 브리핑하는 완료 보고서를 상신합니다.
   - 완료 보고서는 필수 4대 표준 서식(① 반영 내역 요약, ② 반영 결과 요약, ③ 검증 계획 요약, ④ 검증 결과 요약)을 반드시 준수하여 작성하며, `revision-archiver` 스킬로 `안티그래비티\walkthrough\` 폴더에 아카이빙합니다.

## 2. 다단계 승인 배포 프로토콜 (MANDATORY)
에이전트는 배포 확인의 3단계 물리적 순서를 무조건 이행해야 합니다.
1. **배포 순서**: **[로컬웹 확인] ➡️ [테스트웹 확인] ➡️ [메인웹 확인]**의 3단계 물리적 순서를 고수합니다.
2. **단계별 승인 잠금**: 각 확인 단계마다 전민재 차장님의 **명시적 승인 오더(예: "로컬웹 확인 완료, 테스트웹 올려줘")**가 대화창에 기록되기 전까지, 에이전트는 `git push` 등 원격 배포 반영 명령을 단독으로 절대 실행할 수 없습니다.

## 3. [조항 7] 명시적 실행 승인 및 AI 행동 통제 제약
- **명시적 승인 키워드 필수 프로세스**: 사용자가 입력창에 **"승인"**, **"허가"**, **"진행"**, **"시작"**, **"Go"** 등의 명시적인 의사 표현을 주어 결재하지 않은 경우, 코드를 수정(`write_to_file`)하거나 원격 반영(`git push`)하는 모든 물리 도구 호출을 **원천 차단**합니다.
- **예외**: 단순 조회(`view_file`, `list_dir`, `git status`)를 통한 현황 진단은 팩트 파악을 위해 즉시 기동하되, 기획안 승인 전까지는 코드의 변경을 유발하지 않습니다.
- **대기 상태 준수**: 비동기 태스크 구동 완료 시, 후속 작업을 임의로 개시하지 않고 완료 상태만 텍스트 보고한 후 즉시 대기합니다.
