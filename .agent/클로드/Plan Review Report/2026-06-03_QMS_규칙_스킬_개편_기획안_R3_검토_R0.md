## 4. 세부 분석

### [I] 이미지 소스 경로에 세션 고유 UUID 하드코딩 (🔴 높음 — 신규)

**어디가:** §2 `[NEW] image-archiver/SKILL.md` → "2. 실행 프로토콜" 1번 소스 경로(line 155).

```
C:\Users\mjjeon\.gemini\antigravity-ide\brain\c0a990f2-3503-48d8-b7d6-603cb3dd318d\
```

**어떻게 문제인가:** 경로 끝의 `c0a990f2-...`는 **현재 대화/세션에만 유효한 brain 세션 UUID**입니다. 세션이 바뀌면 이 디렉토리명도 바뀌므로, 하드코딩하면 다음 작업부터 소스 폴더를 찾지 못해 복사가 100% 실패합니다. 더 근본적으로, 이 기획안이 AS-IS에서 비판했던 "물리 경로 하드코딩으로 인한 유지보수성 취약점"을 **신설 스킬에서 그대로 재현**하는 자기모순입니다.

**대책방안:** 소스 경로를 고정값이 아니라 **동적으로 해석**하도록 명세합니다. 예를 들어 `brain\` 하위에서 **가장 최근 수정된 세션 폴더**를 선택하거나, 그 안에서 **방금 생성된(최신 mtime) 이미지 파일**을 글롭으로 특정하는 방식이 안전합니다. UUID 부분은 와일드카드/최신 폴더 탐색으로 대체하고, 절대 리터럴 UUID를 박지 않습니다.

### [J] 이미지 복사가 에이전트 호출 의존 — 강제·자동 미보장 (🔵 낮음 — B 잔여)

**어디가:** §1 `[조항 6]`(line 70), `image-archiver` 동작 원리(line 152).

**어떻게 문제인가:** R2 대비 "에이전트 스스로 즉시 호출"로 문구를 현실에 맞게 낮춘 점은 타당합니다. 다만 이는 여전히 **에이전트가 매번 호출을 기억해야** 동작하며, 누락 시 백업이 안 됩니다. 조항 6의 "즉시 강제"라는 표현과 실제 보장 수준(베스트 에포트) 사이에 간극이 남습니다.

**대책방안:** 현 수준으로도 운영은 가능하나, "강제"를 진짜 보장하려면 워크스페이스의 `.agent/hooks/`에 `generate_image` 후처리 훅을 거는 방안을 후속 과제로 검토 권고. 당장은 조항 6 문구를 "생성 직후 에이전트가 호출한다(누락 방지 권고)" 수준으로 일치시키면 충분합니다.
