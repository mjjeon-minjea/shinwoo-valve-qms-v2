## 3. 목표 상태 (TO-BE) — 차장님 의도 (반드시 이 정의대로 구현)

### 3-1. 3계층 역할 정의

| 계층 | 역할 | 데이터 연동 |
|------|------|-------------|
| **로컬** | 순수 개발 전용 | **라이브 직접 연동 없음.** 필요 시 `db:pull`로 데이터를 `db.json`에 *땡겨오는* 방식만 허용 |
| **테스트웹** | 로컬 개발물 배포·시험·점검 | **staging Supabase**(`srzaanvojyhwzugoaimk`) 연동. 테스트·점검 시 **메인 Supabase 데이터를 staging으로 가져와** 현실적 검증 |
| **메인웹** | 실제 사용하는 무결점 prod | **메인 Supabase**(`zuahpjdsypovxdplxryw`) 연동. 로컬·테스트웹과 **완전 별도 운영** |

### 3-2. 승급 파이프라인 (승인 게이트)

```
[로컬]  --(차장님 확인 + "승인")-->  [테스트웹: staging 브랜치]  --(차장님 확인 + "승인")-->  [메인웹: main 브랜치]
```

- 모든 단계 전이는 **차장님의 명시적 "승인" 1건**을 전제로 한다. 자동 승급 금지.
- 코드는 **브랜치로 분리**한다: `staging` 브랜치 → 테스트웹, `main` 브랜치 → 메인웹.
- 데이터 흐름은 **메인 → staging 단방향 복제(read)** 만 허용. 역방향(staging/로컬 → 메인 직접 쓰기) 금지.

## 6. 제약 및 절대 금지 사항 (안티그래비티 대상)

1. **증빙 강제**: 모든 완료·상태 주장은 *실제 명령 출력·대시보드 캡처·파일 diff*로만 한다. 기억·문서·KI를 근거로 들면 그 보고는 **무효**.
2. **prod 데이터 보호**: 메인 Supabase(`zuahpjdsypovxdplxryw`)에 **직접 쓰기/덮어쓰기 절대 금지.** 데이터 흐름은 메인→staging 단방향 read만.
3. **미검증 완료 선언 금지**: push·설정·배포가 실제 반영됐는지 확인(`git branch -a`, `vercel ls`, 대시보드) 전에는 "완료"라 하지 않는다.
4. **단계 전이 사전 보고**: 각 승급 직전 "무엇을 / 어디로(정확한 repo·브랜치·도메인) / 어떤 명령으로 / 롤백은 어떻게"를 보고하고, **차장님 "승인" 후에만** 실행한다.
5. **`.gemini` KI 비간섭**: `staging_status.md` 등 stale KI는 **차장님이 직접 삭제**한다. 안티는 손대지 말되, **해당 KI 자동 생성이 아직 활성인지 확인해 보고**한다.
6. **상태 라벨링**: 모든 보고에 "완료(증빙) / 진행중 / 예정 / 추정"을 명시한다.
