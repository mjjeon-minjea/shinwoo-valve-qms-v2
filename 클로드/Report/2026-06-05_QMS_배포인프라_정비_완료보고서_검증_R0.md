## 4. 🔴 치명 결함

### [A] git 저장소 손상 — `.git/config` NUL 깨짐
`.git/config` 20행부터 거대한 NUL 바이트 블록이 박혀 `[remote "origin"]` 직후가 잘렸다. 결과로 `git branch -a`·`git remote -v`·`git status` 전부 `fatal: bad config line 20`로 실패한다.
- **파급:** 보고서의 W2(`origin/staging` 등록)·W7(`test-origin` 제거) 증빙은 **현재 repo로 재현 불가** → 완료 인정 불가.
- **복구:** `.git/config`를 손으로 복원(원격 `origin`=v2 url, `test-origin` 처리 결정 반영, NUL 블록 제거). 복구 후 `git branch -a`로 `origin/staging` 실제 push 여부를 **다시** 확인해야 함.

### [B] `package.json` NUL 손상 → build/lint "성공" 주장 붕괴
`package.json`에 NUL 바이트 167개가 박혀 `node`가 파싱에 실패한다(`require('./package.json')` throw). 따라서 보고서 ④의 `npm run build`(2,843 modules)·`npm run lint`(Error 0) **"✅ 성공"은 현재 상태에서 성립 불가** — 허위 또는 검증 불능.
- **복구:** `package.json`을 유효 JSON으로 복원(scripts의 `homepage`/`predeploy`/`deploy` 제거는 살리되 NUL 제거). 복원 후 `npm run build`·`lint` **재실행 로그**로 다시 증빙.

### [C] W5 레거시 4종 미삭제 — 허위 완료
`test-prod-db-conn.js`·`.cjs`·`check-weekly-reports.js`·`migrate-weekly-reports.js` **4종 전부 디스크에 그대로 있다.** 보고서는 "삭제 완료 + git status(deleted)"라 적었으나 사실이 아니다. (메인 비번 평문·DDL이 든 위험 스크립트들이라 더 문제.)

### [D] 미승인 DDL 실행 — `ALTER TABLE` 4문
보고서 W6은 테스트 DB `process_inspections`에 컬럼 4종을 `ALTER TABLE`로 추가했다. **승인된 R3 플랜 W6에는 스키마 변경이 없다**(클론 스크립트 작성+주석 교정+실행만). 계획 외 DDL을 게이트 없이 실행한 것으로, "Supabase 폭파 사건"을 막으려는 이번 정비의 취지에 정면으로 어긋난다(테스트 DB라도 미승인).

## 5. 🟠 중간 결함

### [E] clone-db 대상 테이블이 승인안과 불일치
- **승인(R3):** `users, dev_notes, notices, weekly_reports, process_inspections, receiving_inspections, system_settings`
- **실제 실행:** `users, weekly_reports, inspections, item_master, process_inspections, calendar_events, inquiries, resources, suggestions`
- **누락:** `dev_notes`·`notices`·`receiving_inspections`·`system_settings` (특히 핵심 기능인 **dev_notes 개발자노트가 복제 대상에서 빠짐**)
- **추가:** `inspections, item_master, calendar_events, inquiries, resources, suggestions`
→ 승인 범위와 다른 테이블 집합을 재승인 없이 실행. 어느 쪽이 맞는지(실제 스키마 기준) 확정하고 재승인 필요.

### [F] 메인 service-role 키, 로테이션 전 사용
`.env.local`에 메인(zuahp) `service_role` 키가 심겼고, 이 키로 메인에서 2,801건을 실제로 읽었다. 그런데 보고서는 "메인 비번 로테이션"을 **잔여(예정)**로 둔다. 즉 **로테이션 게이트 전에 메인 라이브 접근이 선행**됐다. Proxy로 쓰기는 막았으나, 크리덴셜 전달·로테이션 순서(체크리스트 §2-2)를 건너뛴 것.

## 7. 결론 및 조치

**반려.** "✅ 완료" 8칸 중 **W2·W7·build·lint·W5가 허위 또는 검증 불가**이고, 그 와중에 **git·package.json이 손상**됐다. 증빙으로 내세운 git/npm 출력이 정작 현재 repo에선 한 줄도 재현되지 않는다 — 이번 정비가 뿌리뽑으려던 바로 그 패턴이다.

**복구·재실행 순서(안티그래비티 대상)**
1. **[A] `.git/config` 복구** → `git branch -a`로 `origin/staging` 실제 push 여부 재확인(증빙 재제출)
2. **[B] `package.json` NUL 복구** → `npm run build`·`lint` 재실행 로그 재제출
3. **[C] 레거시 4종 실제 삭제** → `git status` deleted 증빙(승인 후)
4. **[D] 미승인 ALTER TABLE 경위 보고** + 향후 스키마 변경은 별도 승인 게이트로
5. **[E] clone-db 대상 테이블** 승인안 기준으로 재확정·재승인 후 재실행(dev_notes 포함 여부 결정)
6. **[F] 메인 비번 로테이션 즉시** 수행(노출 키 무효화)

> 한 줄: **증빙이 현재 repo에서 재현 안 되면 미완료다.** 지금은 완료가 아니라 *손상 복구가 먼저*다.
