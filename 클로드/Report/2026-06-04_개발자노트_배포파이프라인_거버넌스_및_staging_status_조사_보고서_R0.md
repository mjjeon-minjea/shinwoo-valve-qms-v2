`PostApproval.jsx`의 배포 상태 뱃지는 3단계처럼 보이나, 실제 트랜지션은 2개뿐이라 명명이 현실과 어긋난다.

| 화면 뱃지 | 실제 의미 | 어긋남 |
|-----------|-----------|--------|
| 🔴 로컬 검증중 | db.json `draft`/`rejected` | OK |
| 🟡 테스트망 대기 | db.json `published` (로컬), Supabase 미전송 | "테스트망"이라지만 실제 테스트웹 배포 아님 |
| 🟢 메인서버 라이브 | Supabase `is_synced=true` | "메인서버"라지만 실제는 Supabase 동기화일 뿐, GitHub Pages 배포와 무관 |

→ 차장님이 겪은 "`🟢 메인서버 라이브`인데 git push 전" 모순의 원인. **데이터 동기화 상태**와 **소스코드 배포 상태**가 한 뱃지에 뒤섞여 있다.

## 4. 치명적 문제

🔴 **단일 진실 소스(SSOT) 미사용.**
배포 현황을 판정할 권위 출처는 `db.json`(데이터)·git/GitHub Pages·Vercel 대시보드(코드)인데, 안티그래비티는 이를 무시하고 외부 stale KI를 신뢰했다. SSOT를 만들어도 에이전트가 안 보면 무의미하다.
- **복구/방지**: "배포 상태는 오직 `db.json`(version·is_synced)·`git log`·배포 대시보드에서만 판정한다. 어떤 `.md` 지식 파일도 배포 사실의 근거로 쓰지 않는다"를 규칙으로 강제.

🔴 **테스트웹(스테이징) 계층 실체 부재.**
사용자가 원하는 3단계 흐름의 가운데 칸(테스트웹)이 저장소에 정의돼 있지 않다(Vercel 설정 없음). 파이프라인을 명문화하기 전에 **테스트웹을 무엇으로 할지(Vercel 신규 연결 vs GitHub Pages 별도 브랜치 vs 로컬 프리뷰)부터 확정**해야 한다.

## 5. 권고 — 승인 게이트 파이프라인 거버넌스

### 5-1. 목표 파이프라인 (차장님 요구안 반영)

```
[로컬]  --(차장님 육안 확인 + "승인")-->  [테스트웹]  --(차장님 육안 확인 + "승인")-->  [메인웹]
```

- **모든 단계 전이는 차장님의 명시적 "승인" 1건을 전제**로 한다. 자동 승급 금지.
- **데이터(개발자 노트)와 소스코드를 분리**해 각각 동일한 3단계 게이트를 적용한다(현재 한 뱃지에 섞인 것을 분리).
- 각 게이트에서 안티그래비티는 전이 **직전에 "무엇을 / 어디로 / 어떤 명령으로" 보낼지 보고**하고, 차장님 승인 후에만 실행한다.

### 5-2. 단계별 게이트 정의(초안)

| 단계 | 데이터(콘텐츠) | 소스코드 | 승인 주체 |
|------|----------------|----------|-----------|
| G1 로컬 검증 | db.json `draft`→`published` | 로컬 빌드/린트 통과 | 차장님 |
| G2 → 테스트웹 | (테스트 DB/스테이징 환경 동기화) | 테스트웹 배포(미정 인프라) | 차장님 명시 승인 |
| G3 → 메인웹 | Supabase Production upsert | `git push` → GitHub Pages 배포 | 차장님 명시 승인 |

※ G2의 테스트웹 인프라가 미정이므로, 이 표는 인프라 확정 후 코드/규칙에 반영해야 함.

### 5-3. 규칙으로 정할 것 vs 스킬로 만들 것

차장님 표현대로 "규칙이냐 스킬이냐" — **둘 다, 역할을 나눠서** 권고한다.

- **규칙(`.agent/rules/`에 신설, 예: `03_deploy_gate.md`)** — *판단·금지·승인 조건* 명문화.
  - SSOT 판정 규칙(§4), 단계별 승인 게이트 필수, 자동 승급 금지, 각 단계 목적지 사전 보고 의무, stale KI 인용 금지.
  - GEMINI.md 최상위 규칙에서 이 조항을 참조하도록 연결(기존 DNAS 프로세스와 동일 방식).
- **스킬(예: `deploy-gate` 또는 기존 `release-sync` 확장)** — *반복 실행 절차* 표준화.
  - 전이 직전 "현재 단계 / 대상 항목 / 목적지(URL·저장소) / 실행 명령 / 롤백법"을 정형 출력 → 차장님 승인 입력을 받는 체크리스트.
  - 배포 후 SSOT(`db.json`·git)에 상태를 기록.

규칙은 "지켜야 할 선", 스킬은 "그 선을 지키게 하는 실행 도구"로 분리하면, 안티그래비티가 임의 판단으로 단계를 건너뛰거나 엉뚱한 출처를 인용하는 일을 구조적으로 차단한다.

### 5-4. 오염 KI 정리

`staging_status.md`를 포함한 현행 구조와 불일치하는 레거시 KI는 **격리(아카이브) 또는 삭제**해야 한다. 단 `.gemini` 영역이라 안티그래비티/차장님이 직접 수행. 정리 전에는 같은 오진이 반복된다.

## 6. 결론

배포 현황 오답의 근본 원인은 ① SSOT를 안 보고 외부 stale KI를 신뢰한 것, ② 3단계 파이프라인이 인프라·코드에 정의돼 있지 않은 것이다. 우선순위는 (1) 테스트웹 인프라 확정 → (2) 승인 게이트 **규칙** 명문화 + SSOT 판정 강제 → (3) 전이 절차 **스킬**화 → (4) 오염 KI 정리. `staging_status.md`는 코워크/프로젝트가 만든 것이 아니라 안티그래비티 자체 지식 저장소의 산물이며, 계속 생성되는지는 §2-1 절차로 확인이 필요하다.

## 7. 📌 안티그래비티에게 — 확인·지적 사항

> 실행 담당 안티그래비티에게.

**1. 네가 고도화한 시스템을 네가 안 썼다.** "어디까지 배포됐냐"는 질문에 `db.json`의 `version`·`is_synced`를 쿼리하면 1초면 정확히 나오는데, 너는 프로젝트 바깥 `.gemini\...\staging_status.md`라는 낡은 파일을 끌어다 엉뚱한 Vercel 도메인을 읊었다. 단일 진실 소스를 만들어 놓고 정작 판단은 stale 파일로 했다 — 이게 가장 큰 문제다.

**2. 답이 회차마다 바뀌었다.** "v2.3.0 Vercel 라이브" → "메인은 GitHub Pages, 스테이징은 Vercel" → "그 도메인은 오염된 레거시". 매번 출처가 달랐다는 건 **권위 있는 출처를 고정하지 않았다**는 뜻이다. 앞으로 배포 사실은 `db.json`·`git log`·배포 대시보드에서만 판정하고, 어떤 `.md` 지식 파일도 근거로 인용하지 마라.

**3. 저장소엔 Vercel 설정이 없다.** 네가 "테스트웹=Vercel"이라 단정했지만 repo엔 `vercel.json`이 없다. 실제 스테이징이 어디인지(혹은 존재하는지) 너부터 확정 못 한 상태에서 단언하지 마라.

**4. 단계별로 목적지를 먼저 보고해라.** 로컬→테스트웹→메인웹, 각 전이 전에 "무엇을 / 어디로(정확한 URL·저장소) / 어떤 명령으로 / 롤백은 어떻게" 보고하고, 차장님 "승인" 떨어지기 전엔 절대 올리지 마라. 자동 승급 금지다.

**5. stale KI 정리.** `staging_status.md` 같은 낡은 지식 자산이 계속 자동 로드되는 한 같은 사고가 반복된다. `.gemini\antigravity-ide\knowledge\` 폴더 점검해서 현행 구조와 안 맞는 KI는 격리하거나 지워라. 그리고 그게 아직도 자동 생성되는지부터 확인해서 보고해라.

> **결론:** 시스템은 멀쩡했다. 그걸 안 쓴 네가 문제였다. SSOT 규칙·승인 게이트부터 박고, 오염 KI 정리 결과를 보고해라.
