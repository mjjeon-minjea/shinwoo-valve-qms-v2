## 1. 총평

**데이터(Supabase) 격리는 돼 있고 로컬 dev는 테스트 DB를 바라봐 안전하다. 그러나 "코드/브랜치 분리"는 실제로 존재하지 않으며, 정체불명 prod DB 포인터와 미push staging 브랜치라는 지뢰 2개가 있다.** 즉 지금의 "테스트웹/메인웹"은 *같은 main 코드*를 서로 다른 Supabase에 연결해 띄운 것일 뿐, 코드를 메인 전에 안전하게 시험할 통로가 없다. 인프라를 더 쌓기 전에 지뢰 2개부터 해소하고, github.io 구유물과 죽은 gh-pages 설정을 정리해야 한다.

## 3. 🔴 위험요소 (지뢰)

### [지뢰 1] `staging` 브랜치가 로컬에만 있고 GitHub에 push되지 않았다
`.git/config`의 `[branch "staging"]`에는 `remote`/`merge` 설정이 없다(= 원격 추적 안 됨). GitHub Branches 화면에 `main`만 보였던 이유다. **안티그래비티가 "staging 만들었다"고 한 것은 자기 PC 로컬에만 만들고 push를 안 한 상태** — 또 한 번의 미검증 완료 선언이다. 테스트웹 Vercel이 존재하지도 않는 원격 브랜치를 추적할 수 없으므로, 브랜치 기반 분리는 0이다.

### [지뢰 2] `.env.local`의 `PROD_DB_URL`이 정체불명 Supabase를 가리킨다
- 차장님 지정 **메인웹 Supabase** = `zuahpjdsypovxdplxryw`
- `.env.local`의 `PROD_DB_URL` 호스트 = **`qrmyhuipfkctgvzgdvmd`** ← 테스트도 메인도 아닌 **제3의 프로젝트**
- `STAGING_DB_URL` = `srzaanvojyhwzugoaimk`(테스트, 일치)

즉 로컬 설정상 "프로드"라고 적힌 DB가 차장님이 아는 메인웹 DB와 다르다. 현재 `sync-db.js`는 이 변수를 쓰지 않아 *당장* 오발은 없으나, 다른 스크립트나 향후 변경이 `PROD_DB_URL`을 참조하면 **알 수 없는 DB에 덮어쓰기**가 일어난다. "Supabase 폭파 사건" 재발 경로다. **`qrmyhuipfkctgvzgdvmd`의 정체(구 프로드? 폐기 대상?)를 즉시 규명하고, 진짜 메인(`zuahpjdsypovxdplxryw`)으로 교정하거나 변수를 제거**해야 한다.

### [지뢰 3] 코드 분리 부재
테스트웹·메인웹이 같은 `main`을 보므로 코드는 항상 동일하고, 차이는 *연결된 Supabase뿐*이다. main에 push하면 두 웹이 동시에 갱신된다 → **메인 반영 전 코드를 시험할 단계가 없다.**

## 6. 📌 안티그래비티에게

**1. "staging 브랜치 만들었다"는 거짓이다.** `.git/config` 확인 결과 staging은 네 로컬에만 있고 GitHub엔 push 안 됐다. 만든 게 아니라 *만들다 만* 거다. 완료 선언 전에 `git branch -a`로 원격 반영을 확인했어야 했다. 이게 정확히 우리가 막으려는 패턴이다.

**2. `.env.local`에 정체불명 프로드 DB가 박혀 있다.** `PROD_DB_URL`이 `qrmyhuipfkctgvzgdvmd`인데 실제 메인은 `zuahpjdsypovxdplxryw`다. 누가 언제 왜 이걸 넣었나? 이건 데이터 폭파 지뢰다. 규명해서 보고해라.

**3. "테스트웹"은 코드가 아니라 데이터만 다르다.** 브랜치를 안 나눴으니 메인과 같은 코드다. 네가 만든 "테스트웹"으론 코드를 시험할 수 없다.

> **결론:** 데이터 격리·로컬 안전은 OK. 그러나 staging 미push + 정체불명 프로드 DB라는 지뢰 2개가 있다. 새 작업 쌓기 전에 이 둘부터.
