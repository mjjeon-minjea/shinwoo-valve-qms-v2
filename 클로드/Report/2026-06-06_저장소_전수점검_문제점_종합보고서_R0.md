## 1. 총평

**🔴 반려 수준의 보안 결함 1건(시크릿 git 커밋)을 포함해, 저장소 위생 문제가 다수 발견됨.** 코드 자체는 `process.env` 기반으로 비교적 양호하나, **시크릿·데이터 덤프·생성 캐시가 git 추적**되고 있고 `.gitignore`에 구멍이 있다. 즉시 조치가 필요한 것은 **시크릿 키 회전 + 히스토리 제거**이며, 나머지는 정리·하이진 과제다.

**점검 방법(신뢰성):** 코워크 마운트 뷰는 staleness(파일 NUL 패딩·readdir/stat 불일치)가 재확인되어, 본 점검은 가능한 한 **git object store(`git ls-tree`/`cat-file`/`grep`/`log`)** 기준으로 교차검증했다(마운트 비의존). 단정이 불가한 항목은 §6에 "검증 보류"로 분리했다.

## 5. 세부 분석

### A. `.gitignore` 구멍 (시크릿 누출의 근본 원인)

현재 `.gitignore`는 `.env`, `.env.local`, `.env.production`, `.env*.local`, `.vercel`를 무시하지만 **다음 변형을 못 막는다**: `.env.local.bak`(`.bak`), `.env.vercel`, `.env.preview`, `.env.production.vercel`. 권고 추가 패턴:

```gitignore
# Env (변형 전체 차단)
.env*
!.env.example
*.bak
# 데이터 덤프 / 백업
db.json
dev_notes_rows.json
*_backup.json
# 빌드 로그
build*.log
build*.txt
# 생성 캐시/그래프 산출물
graphify-out/
# 스크래치
scratch/
```

> 주의: `db.json`이 런타임에 실제로 쓰이는 데이터면 `.gitignore` 대신 별도 데이터 관리가 필요. 코드 참조 여부를 안티가 확인 후 결정.

### B. 생성 캐시·데이터 덤프 비대화

- `graphify-out/` = **771 추적파일**(이 중 `cache/` 765개 = AST 캐시). `manifest.json` 244KB, `cache/stat-index.json` 209KB. 전형적 **생성물 → 커밋 대상 아님**. churn·diff 노이즈·리포 비대 유발.
- 루트 `db.json`(753KB), `dev_notes_rows.json`(156KB), `src/data/mes_process_inspections.json`(406KB) = 데이터 덤프. 코드 의존 없으면 추적 해제 권장.
- 결과: **`.git` 62MB**(소형 앱 치고 큼) — 위 생성물·이미지·`package-lock`이 히스토리에 누적된 영향.

### C. `scripts/` 데이터 백업 (PII 우려)

`staging_users_backup.json`, `staging_weekly_reports_backup.json` = **운영 DB의 사용자/보고 데이터 스냅샷**. 사용자 개인정보(이름·계정 등) 포함 가능 → git 커밋은 개인정보 보관 정책상 부적절. 추적 해제 + 히스토리 검토 권고.

### D. 소스 `localhost` 하드코딩

`src/components/PostApproval.jsx`(라인 32·76·123·172 등), `DevNotes.jsx`(라인 41)에서 `http://localhost:3001/dev_notes`를 직접 fetch. 코드 주석상 **"승인/동기화는 로컬 인트라넷에서만"** 의도된 설계로 보이나:

- 같은 URL이 **6곳 이상 하드코딩** → 포트/호스트 변경 시 누락 위험. **설정 상수/`.env`(예: `VITE_LOCAL_API`)로 중앙화** 권고.
- 단, `DevNotes.jsx`의 **읽기(GET)까지 localhost** 고정이면 배포(Vercel)에서 노트가 안 보일 수 있음 → 의도 확인 필요(검증 보류).

### E. 중복·정리 대상

- `HANDOFF.md`: 루트(6,713B) vs `shinwoo-system/HANDOFF.md`(3,136B) — 동일 제목, 다른 내용. **단일 진실 1곳 + 나머지 링크** 권고.
- `scratch/clean_obsidian_wiki.cjs`(7,055B) ↔ `clean_obsidian_wiki.js`(5,910B) — 동일 목적 2버전.
- `.gitattributes`(`eol=lf`) 영향으로 워킹트리 CRLF 파일이 대량 `M`(수정)으로 잡히는 churn 관측(앞선 R2 점검 시 141건). 의도된 정규화면 1회 커밋으로 흡수 권고.

## 6. 검증 한계 (코워크가 확정 못 한 항목 — 실터미널 교차확인 필요)

1. **`.git` 인덱스/리플로그 손상**: `git status`가 *"index uses … extension, which we do not understand"*, `git fsck`가 *"invalid reflog entry"* 반환. 단 **object store 명령(`ls-tree`/`cat-file`/`log`/`grep`)은 전부 정상** → **마운트 읽기 아티팩트일 가능성이 높음**. 그러나 안티 실터미널에서도 `git status`가 같은 오류면 **실제 인덱스 손상**이므로 인덱스 재생성(`git read-tree HEAD` 또는 `.git/index` 재빌드) 필요. *(코워크는 수정하지 않음)*
2. **마운트 desync 일반**: 파일 NUL 패딩(예: `GEMINI.md` 마운트 14,342B 중 3,054가 NUL, 실 blob 11,288B), `_shared` 삭제 여부 ls↔git 충돌 등 → 본 보고서의 디스크 수치는 **git 기준**을 우선했으나, 최종 확정은 실터미널 권장.
3. `api/admin-update-member.js`의 인증/권한 가드 적정성은 코드 정독 범위를 넘어 **별도 시큐리티 리뷰** 권고.
