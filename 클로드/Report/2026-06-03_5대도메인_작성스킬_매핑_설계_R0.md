## 1. 배경 및 목적

현재 규칙은 5대 도메인(Plan·Task·Walkthrough·Report·images)에 대해 **버전 관리(revision-archiver)·복사(image-archiver)·동기화(release-sync)·검사(plan-self-review)** 만 스킬로 위임하고 있다. 정작 **"각 문서를 어떤 스킬로, 어떤 형식으로 작성하는가"** 는 지정돼 있지 않아, 에이전트가 자유 형식으로 작성한다. 본 설계는 차장님 지시에 따라, `.agent/skills`의 **기존 범용 스킬을 도메인별로 매핑**하고, 규칙에서 저장경로·한글 DNAS 포맷을 **강제 오버라이드**하는 방식을 정의한다.

## 2. 현황 분석

| 도메인 | 현재 연결 | 작성 스킬 | 문제점 |
|--------|-----------|-----------|--------|
| Plan | plan-self-review, revision-archiver | ❌ 없음 | 작성 형식 비표준, 에이전트 임의 |
| Task | revision-archiver | ❌ 없음 | Step 마킹 형식 비표준 |
| Walkthrough | release-sync, revision-archiver | ❌ 없음 | 완료 보고 형식 비표준 |
| Report | revision-archiver | ❌ 없음 | 분석 보고 형식 비표준 |

`.agent/skills`의 범용 후보들은 외부 도입(superpowers/Anthropic) 계열이라 **저장경로·언어·전제가 QMS와 불일치**한다. 따라서 그대로 쓰지 않고 규칙에서 오버라이드한다.

## 3. 매핑안 (도메인 ➔ 범용 스킬 + 오버라이드)

| 도메인 | 매핑 스킬 | 선정 이유 | 규칙에서 강제 오버라이드할 항목 |
|--------|-----------|-----------|-------------------------------|
| **Plan** | `writing-plans` | 다단계 작업의 구현 계획서 작성 전용 | 저장경로 `docs/superpowers/plans/` ➔ `안티그래비티\plan\` / 영어·TDD·커밋 전제 무시, 한글 DNAS `implementation_plan` 형식 / 작성 후 `plan-self-review` ➔ `revision-archiver` 순으로 연계 |
| **Task** | `planning-with-files` | 작업을 Step으로 분해·진행 추적(task_plan/progress) | 산출물명·경로 ➔ `안티그래비티\task\` / 자체 hook(.planning/) 동작 주의, QMS task 규격으로 출력 |
| **Walkthrough** | `internal-comms` | 완료·상태·결과 보고(3P/프로젝트 업데이트/인시던트) 포맷 보유 | 저장경로 ➔ `안티그래비티\walkthrough\` / 한글 완료보고 형식 / 작성 후 `release-sync` 연계 |
| **Report** | `doc-coauthoring` | 분석·제안·기술/결정 문서 협업 작성 워크플로 | 저장경로 ➔ `안티그래비티\report\` / 한글 분석보고 형식 |

> images는 작성 대상이 아니므로 기존 `image-archiver`(복사) 유지.

## 4. 규칙 수정 문안 (02_dnas_process.md §4 제안)

기존 §4의 "관리·동기화" 위임에 더해, **작성(authoring) 위임 매핑**을 명시한다. 아래는 §4에 추가/교체할 제안 문안이다.

```markdown
## 4. 스킬 실행 안내 및 자동 위임

### 4-1. 도메인별 작성(Authoring) 스킬 — 작성 시 반드시 호출
- 본 프로젝트의 문서 4종은 작성 시점에 아래 전용 스킬을 호출하여 작성한다.
  단, 각 스킬의 기본 저장경로·언어·전제는 무시하고, 아래 QMS 규격을 강제 우선한다.
  - **Plan (기획안)** ➔ `writing-plans` 스킬 / 저장 `안티그래비티\plan\` / 한글 implementation_plan 형식
  - **Task (작업 명세)** ➔ `planning-with-files` 스킬 / 저장 `안티그래비티\task\` / 한글 task Step 형식
  - **Walkthrough (완료 보고)** ➔ `internal-comms` 스킬 / 저장 `안티그래비티\walkthrough\` / 한글 완료보고 형식
  - **Report (특별 분석 보고)** ➔ `doc-coauthoring` 스킬 / 저장 `안티그래비티\report\` / 한글 분석보고 형식

### 4-2. 작성 후 관리·동기화 위임 (기존 유지)
  - Plan/Task/Walkthrough/Report 리비전 관리 ➔ `revision-archiver` 스킬
  - 이미지 자동화 보관 ➔ `image-archiver` 스킬
  - 릴리즈 인트라넷 동기화 ➔ `release-sync` 스킬
  - 기획안 제출 전 자가검토(필수) ➔ `plan-self-review` 스킬

### 4-3. 표준 연계 순서
  - Plan: writing-plans 작성 ➔ plan-self-review(High 0건) ➔ revision-archiver(버전)
  - Task: planning-with-files 작성 ➔ revision-archiver
  - Walkthrough: internal-comms 작성 ➔ revision-archiver ➔ release-sync(동기화)
  - Report: doc-coauthoring 작성 ➔ revision-archiver
```

## 5. 주의사항 (매핑 방식의 리스크)

1. **경로 충돌**: `writing-plans`는 `docs/superpowers/plans/`, `planning-with-files`는 `.planning/` 디렉토리를 기본값으로 쓴다. 규칙 오버라이드만으로는 스킬 본문이 기본 경로로 저장할 위험이 있으므로, **각 SKILL.md 본문의 저장경로 문구도 함께 QMS 경로로 덮는 것을 권고**(매핑만으로는 불완전).
2. **planning-with-files 자체 hook**: 이 스킬은 `UserPromptSubmit/PreToolUse/Stop` 등 다수 hook을 frontmatter에 내장하고 있어 `.planning/` 기반으로 동작한다. Task 도메인에 붙일 경우 hook이 QMS 흐름과 간섭하지 않는지 검증 필요.
3. **언어·전제 불일치**: 세 스킬 모두 영어·소프트웨어 엔지니어링(TDD/커밋/PR) 전제로 서술돼 있어, 한글 QMS 결재 맥락과 톤이 다르다. 오버라이드 문구에 "형식·언어는 QMS 규격 우선"을 명시해야 한다.
4. **대안 참고**: 충돌 관리 부담이 크다고 판단되면, 빈 껍데기인 `qms-orchestrator`(현재 SKILL.md 없음)에 4종 형식을 직접 정의하는 단일 라우터 방식이 더 깔끔할 수 있음(차후 검토 여지).

## 7. 결론

5대 도메인 작성 스킬 미지정은 실재하는 공백이며, 매핑 방식으로 메울 수 있다. 단 **규칙 오버라이드만으로는 불완전**하므로, 매핑 규칙 개정과 함께 **각 스킬 SKILL.md의 경로·형식 보정**을 반드시 병행해야 한다(주의사항 1·2). 본안은 §4 개정 + 스킬 4종 보정의 2단 작업으로 진행하길 권고한다.
