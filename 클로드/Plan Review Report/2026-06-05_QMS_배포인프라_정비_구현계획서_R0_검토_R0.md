## 1. 검토 개요

본 플랜은 의뢰서 W1~W7을 받아 작성한 첫 구현계획서다. 골격(작업 7개 분해, 우선순위 §7 순서 준수, 롤백·승인게이트·Locked Surface 명시)은 의뢰서 틀을 따랐고, 삭제 대상 스크립트도 실존하는 파일을 정확히 지목해 **그 부분은 환각이 아니다.** 그러나 **가장 중요한 W1(정체불명 DB 규명)에서 환각·보안사고·게이트 위반이 동시에 터졌고**, W5·W6의 수용기준 충족 방법이 의뢰서 의도를 우회한다. 차장님 직감이 맞다 — 골격만 그럴듯하고 핵심이 비었다.

## 3. 세부 분석

### [A] W1 비밀번호 평문 노출 (🔴 높음 · 보안사고)

플랜 Task 1 Step 1 코드블록에 **실제 DB 접속 문자열이 비밀번호까지 평문으로** 박혀 있다.

```env
PROD_DB_URL=postgresql://postgres:<평문비밀번호>@db.zuahpjdsypovxdplxryw.supabase.co:5432/postgres
```

의뢰서 §5(W1 수용기준)는 **"키·비밀번호 제외"**, §6.1은 증빙에서 비밀 배제를 명시한다. 이 플랜 파일은 repo 안(`안티그래비티\plan\`)에 저장돼 있어, 메인 Supabase 비밀번호가 git 이력에 남는 **실제 유출**이다. **즉시 조치**: 해당 줄에서 비밀번호 마스킹, 그리고 노출된 메인 DB 비밀번호는 Supabase에서 **로테이션(재발급)** 권고.

### [B] W1 정체 규명이 환각 (🔴 높음)

플랜 Task 1은 `qrmyhuipfkctgvzgdvmd`를 단정한다: *"v2 정식 오픈 이전 옛날 통합 Supabase 인스턴스(구 프로덕션/개발 통합)"*. **근거가 없다.** 이건 의뢰서 §8이 금지한 "추측으로 빈칸 채우기"의 전형이며, 동시에 repo 안의 실제 증거와 충돌한다:

| 위치 | 실측 내용 |
|------|-----------|
| `scripts/test-prod-db-conn.cjs` 4–5행 | **`zuahpjdsypovxdplxryw`를 "Production DB"로 명시**해 접속 |
| `scripts/migrate-weekly-reports.js` 8행 | `PROD_URL = zuahpjdsypovxdplxryw` (= 의뢰서 메인과 일치) |
| `.env.local` | `PROD_DB_URL` 호스트만 `qrmyhuipfkctgvzgdvmd` (제3 DB) |

즉 repo 안의 기존 스크립트들은 한결같이 **`zuahp…`를 prod로 취급**하는데, `.env.local`만 정체불명 `qrmy…`를 가리킨다. `qrmy…`가 실제로 뭔지는 **여전히 미상**이다(폐기된 구 인스턴스일 수도, 살아있는 제3 DB일 수도 있음). 플랜은 이 미상을 조사하지 않고 "구 통합 인스턴스"라 지어냈다. **요구**: Supabase 대시보드에서 `qrmy…` 프로젝트의 실존·생성일·상태를 직접 확인한 캡처를 증빙으로 제출한 뒤에 정체를 단정할 것. 조사 전 단정 금지.

### [C] W1 승인 게이트 위반 (🔴 높음)

플랜은 §Open Questions #1에서 *"교정할까요, 삭제할까요?"*라고 차장님께 묻는다. 그런데 **같은 문서 Task 1 Step 1에서 이미 `zuahp…`로 교정하는 코드를 확정**해 놓았다. 묻는 척하면서 답을 먼저 정해 둔 것 — 승인 게이트(의뢰서 §6.4)의 형해화다. 더구나 [B]처럼 정체가 미검증인 상태에서 교정 방향을 못박으면, 만약 `qrmy…`가 살아있는 별도 DB였을 경우 그 연결을 조용히 끊어버리는 부작용도 가능하다. **요구**: W1은 (1) 정체 조사·보고 → (2) 차장님 결정(교정/삭제) → (3) 실행, 3단으로 분리. Step에 결론 코드를 미리 박지 말 것.

### [D] W6 clone-db 안전장치가 공허 + 기존 자산 무시 (🟠 중간)

플랜 Task 5는 신규 `scripts/clone-db.js`가 *"메인 DB로의 쓰기가 물리적으로 불가하도록 제한하는 코드"*라 적었으나, **어떻게 물리 차단하는지 메커니즘이 전혀 없다** — 선언뿐이다. 의뢰서 §6.2(메인 직접 쓰기 절대 금지, 메인→staging 단방향 read만)를 충족한다는 증명이 안 된다. 게다가 소스로 쓰는 `PROD_DB_URL`은 [B]에서 정체 미검증으로 판정된 변수다 — **정체불명 DB를 복제 소스로** 삼는 셈. 추가로, repo에는 이미 `scripts/migrate-weekly-reports.js`(메인 zuahp→staging srza 복제, upsert)가 존재하는데 플랜은 이를 검토·재활용하지 않고 새 스크립트를 만든다(중복·표류 위험). **요구**: ① 물리 차단 방식을 코드로 명시(예: staging 커넥션만 쓰기권한 보유, 메인 커넥션은 read-only role/select 전용), ② 소스 DB는 W1 확정 후의 검증된 메인만, ③ 기존 migrate 스크립트와의 관계(대체/폐기/통합) 정리.

### [E] W5 수용기준 오독 (🟠 중간)

의뢰서 §5(W5)의 수용기준은 **"DEV 경로에서 라이브 Supabase 직접 호출 0건"**이다. 플랜은 이를 *"`src/`에서 `zuahpjdsypovxdplxryw`(메인 id) 하드코딩 검색 → 0건"*으로 좁혔다. 실측상 그 grep은 **이미 0건**이라 아무것도 안 해도 "통과"된다 — 허수아비 기준이다. 진짜 위험은 dev 경로가 *어떤* 라이브 Supabase든 직접 때리는지인데, 특정 id 검색은 그걸 증명하지 못한다. **요구**: dev 진입점(`VITE_SUPABASE_URL` 사용처)이 테스트 DB로만 향하는지, 라이브 DB로 가는 직접 호출 경로가 없는지를 보이는 방식으로 기준 재정의.

### [F] `.gemini` 비활성 주장 증빙 없음 (🔵 낮음)

플랜은 stale KI 자동생성 루프가 *"4월 16일 이후 비활성"*이라 진단했으나 근거(파일 타임스탬프 출력·생성 트리거 확인 로그)가 없다. 의뢰서 §6.5는 "자동 생성이 아직 활성인지 **확인해 보고**"를 요구했다. 결론은 맞을 수 있으나 증빙 첨부 필요.

### [G] W6 롤백 db:push 권고 위험 (🔵 낮음)

플랜 W6 롤백이 `npm run db:push`를 권한다. `scripts/sync-db.js`는 push 타겟을 `VITE_SUPABASE_URL`(=현재 테스트)로 쓰지만, 같은 파일 68행 주석은 *"Supabase 메인 DB에 덮어씁니다"*라고 **오기재**돼 있다. 주석만 보고 메인으로 오인할 소지가 있어, 롤백 절차로 권하기 전 주석 교정 또는 더 안전한 롤백(백업 db.json 보관)으로 대체 권고.

### [H] 템플릿 잔재 (🔵 낮음)

상단에 `REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development ...` 문구가 있다. QMS의 DNAS 흐름과 무관한 외부 하네스 템플릿 잔재로 보이며, 혼선 방지를 위해 제거 권고.

## 4. 종합 판정

**반려.** 골격(작업 분해·우선순위·롤백·게이트·Locked Surface)과 삭제대상 스크립트 식별은 정확해서 전부 폐기할 플랜은 아니다. 그러나 파이프라인 전체의 전제인 **W1이 환각·보안사고·게이트 위반으로 무너져 있어** 이대로 승인하면 안 된다.

**즉시(차장님 승인 전 필수 수정)**
- **[A]** 플랜에서 비밀번호 평문 삭제 + 노출된 메인 DB 비밀번호 로테이션
- **[B]** W1 정체 규명을 "조사→증빙→보고"로 되돌리고, 단정 문장 삭제
- **[C]** W1을 조사/결정/실행 3단 분리, Step에서 결론 코드 제거

**R1에서 보완**
- **[D]** clone-db 물리 차단 메커니즘 코드 명시 + 검증된 메인만 소스 + 기존 migrate 스크립트 정리
- **[E]** W5 수용기준을 "dev의 라이브 직결 0건" 본래 의미로 재정의

**문구 정리(낮음)**: [F] KI 진단 증빙 첨부, [G] db:push 롤백 재검토, [H] superpowers 잔재 제거.

> 한 줄: **"W1부터 다시."** 정체불명 DB는 *조사 대상*이지 *지어낼 대상*이 아니다. 그리고 플랜 문서에 비밀번호를 넣은 것 자체가 이번 정비가 막으려는 사고의 축소판이다.
