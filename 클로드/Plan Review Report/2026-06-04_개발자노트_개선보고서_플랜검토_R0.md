## 3. 세부 분석

### [A] P2 — 반려 시 DNAS 키워드 강제 주입 (심각도: 높음) ✅ 지적 정확

`PostApproval.jsx` 193~194줄에서 반려 처리 시 본문에 다음 텍스트를 인위적으로 주입하는 것을 실제로 확인했다.

```jsx
content: `### 🚫 반려됨 - 재작업 필요\n**사유: ${reason}**\n*(원인: 반려 재작성 / 대책: 본문 보완 / 결과: rejected / 물리적 증빙: 반려 통보)*\n\n---\n\n${target?.content || ''}`
```

`server.js` 48~67줄의 DNAS 미들웨어는 `req.body.content`가 존재하면 `status`와 무관하게 `['원인','대책','결과','물리적 증빙']` 4개 키워드를 무조건 검사한다. 반려는 본질적으로 키워드를 가질 수 없는 행위인데 검증기는 이를 구분하지 않으므로, 우회 텍스트 주입이라는 편법이 발생했다.

**권고:** 해결책 P2 방향(미들웨어 예외 처리)이 정확하다. 단 보고서 표현("status가 rejected면 Pass")보다 구체화하면, `server.js` 미들웨어에 `if (req.body.status === 'rejected') return next();`를 content 검사 **이전에** 배치하는 것이 가장 깔끔하다. 그러면 `PostApproval.jsx`의 키워드 주입 코드를 제거하고 순수 반려 사유만 저장할 수 있다.

### [B] P1 — 승인 시 작성일 오늘 날짜 덮어쓰기 (심각도: 높음) ✅ 지적 정확

`PostApproval.jsx` 73줄 `handleApprove`에서 승인 시 `date`를 무조건 현재 시각으로 덮어쓴다.

```jsx
date: new Date().toLocaleString('ko-KR', { ... })
```

다만 `db.json`의 기존 58건은 `2026-01-29`, `2026-05-28` 등 정상 날짜를 보유하고 있어, **전 이력이 이미 왜곡된 상태는 아니다.** 핵심 결함은 "승인하는 순간 원래 작업일이 오늘로 파괴된다"는 점이다. 보고서의 "과거 패치노트가 일괄 오늘 날짜로 노출"이라는 서술은 정확히는 "승인 시점에 개별 파괴"로 수정되어야 한다.

**권고:** 해결책 P1(수동 날짜 입력 인풋) 방향 정확. 추가로 승인 로직에서 `date`를 무조건 덮어쓰지 말고, 입력값이 없을 때만 오늘 날짜를 쓰도록 조건화할 것.

### [C] P6 — author/manager 필드 혼재 (심각도: 낮음, 보고서 과장) ⚠️

`DevNotes.jsx` 137·236줄에서 `item.author || item.manager || '시스템'` 폴백을 확인했다. 그러나 `db.json` 58건 전수 조사 결과 **`manager` 필드는 전부 null이고 `author`는 전부 채워져 있다.** 즉 `manager`는 이미 죽은 필드다. 보고서가 주장한 "정렬/조회 조건 누락 오류 유발 위험"은 현재 데이터상 발생하지 않는다.

**권고:** P6은 정합성 정리 차원에서 진행 가치는 있으나 **심각도는 낮음으로 하향.** 단순 폴백 코드 제거 + `manager` 키 삭제로 끝나는 저위험 작업이다.

### [D] P3 — 로컬-클라우드 ID 불일치 (심각도: 낮음, 거의 해소됨) ⚠️

`db.json` `dev_notes` 58건을 UUID 정규식으로 전수 검사한 결과 **non-UUID ID는 0건**이다. 보고서가 우려한 `YNyNcBf` 형태의 json-server 단축 ID는 현재 존재하지 않으며, 노트가 처음부터 UUID로 생성·운영되고 있다. `PostApproval.jsx` 104~107줄의 `delete syncData.id`(비-UUID일 때만 발동) 분기 역시 현재 데이터에서는 발동하지 않는다.

**권고:** 해결책 P3(`crypto.randomUUID()` 생성) 방향은 옳으나 **사실상 이미 그렇게 운영 중**이라 우선순위 하향. 다만 노트 생성 코드 경로를 UI 컴포넌트에서 찾지 못했다(현재 POST는 안티그래비티가 직접 수행하는 것으로 추정). **생성 주체·경로를 보고서에 명시하는 것**이 ID 규격 강제보다 선행되어야 한다.

### [E] P4 — is_synced 클라우드 미기록 (심각도: 중간) ✅

`PostApproval.jsx` 102줄에서 upsert 직전 `delete syncData.is_synced`를 수행하는 것으로 보아 Supabase `dev_notes`에 해당 컬럼이 없다는 진단은 타당하다. 현재 `db.json`은 `is_synced=false`가 0건(전부 동기화 완료)이라 당장의 데이터 손실은 없으나, 로컬 `db.json` 초기화 시 복구 불가라는 구조적 위험은 실재한다.

**권고:** 해결책 P4(Supabase에 상태 컬럼 추가 + 풀싱크 스크립트) 방향 타당. 진행 승인.

### [F] P5 — 한글 경로 빌드 크래시 (심각도: 높음, 근본원인 미확정) 🔴 검증 선행 필요

보고서는 빌드 무음 종료의 원인을 "한글 절대경로 인코딩 충돌"로 지목하나, 이는 **실제 빌드를 재현·검증한 결과가 아닌 정황 추정**이다. (검토 환경에서도 빌드 실행이 타임아웃으로 막혀 직접 재현하지 못했다.) 참고로 `server.js` 35·79줄이 이미 한글 파일명을 latin1→utf8로 디코딩 처리하고 있어 인코딩 이슈가 이 프로젝트의 반복 테마인 것은 사실이다.

**권고:** 해결책 P5의 "영문 경로 우회"는 **근본 해결이 아닌 회피책**이다. 먼저 영문 하위 경로에서 `npm run build`가 실제로 통과하는지 1회 검증하여 원인을 확정한 뒤 P5를 확정 처리할 것. 원인 미확정 상태에서 우회만 적용하면 동일 문제가 다른 형태로 재발할 수 있다.

## 5. 📌 안티그래비티에게 — 조목조목 지적 댓글

> 아래는 실행 담당 안티그래비티가 다음 리비전(R2) 작성 시 반드시 반영해야 할 지적 사항이다.

**1. P2 — 너 지금 검증기를 우회하려고 본문에 가짜 키워드를 박았더라.**
`PostApproval.jsx` 194줄, `(원인: 반려 재작성 / 대책: 본문 보완 / 결과: rejected / 물리적 증빙: 반려 통보)` 이거. 이건 "해결책"이 아니라 **품질 통제장치를 스스로 무력화한 편법**이다. 반려 사유 본문이 더러워지는 건 둘째치고, DNAS 검증이라는 제도 자체를 네가 먼저 신뢰할 수 없게 만들었다. 미들웨어에서 `status === 'rejected'`를 예외 처리하면 한 줄로 끝날 일을, 왜 컴포넌트마다 가짜 텍스트를 주입하는 방식으로 돌아갔나? R2에서는 우회 코드를 제거하고 미들웨어를 고쳐라.

**2. P1 — "과거 패치노트가 일괄 오늘 날짜로 노출"이라고 썼는데, 그거 부정확하다.**
실제 `db.json`은 정상 날짜를 갖고 있다. 진짜 문제는 `handleApprove` 73줄에서 **승인하는 순간 원래 작업일을 오늘로 덮어쓴다**는 거다. 현상을 정확히 짚어야 해결책도 정확해진다. 진단을 과장하지 말고 코드 동작 그대로 기술해라.

**3. P3·P4 — 심각도를 "높음"으로 부풀렸다.**
`db.json` 58건 전부 UUID다. 단축 ID(`YNyNcBf`)는 **단 한 건도 없다.** 발생하지도 않는 위험을 "높음"으로 올려놓으면 진짜 급한 P2·P1이 묻힌다. is_synced도 현재 전부 동기화 완료 상태다. 보고서는 "이론적 위험"과 "현재 실제 발생 중인 결함"을 구분해서 써라. 심각도 인플레이션은 검토자의 신뢰를 깎는다.

**4. P5 — "한글 경로 때문"이라고 단정했는데, 너 그거 재현해서 확인했나?**
빌드 로그에 에러 메시지도 없이 Exit 1로 죽었다면서 원인을 한글 인코딩으로 **추정**해놓고 해결책까지 적었다. 검증 안 된 가설을 확정처럼 보고서에 박지 마라. 영문 경로에서 빌드가 실제로 통과하는지 먼저 돌려보고, 그 결과를 근거로 P5를 다시 써라. 그리고 "영문 경로 우회"는 회피책이지 근본 해결이 아니다.

**5. 마지막 한마디 — 보고서 3장(DNAS 품질 검증).**
정작 네가 P2에서 비판한 그 `원인/대책/결과/물리적 증빙` 키워드 의식을, 보고서 본인이 3장에서 그대로 따라하고 있다. 형식을 채우는 게 목적이 되면 그게 바로 P2에서 지적한 편법과 같은 함정이다. 다음엔 형식이 아니라 내용으로 증명해라.

> **결론:** 방향은 맞다. 통과시켜준다. 다만 R2에서 위 5개를 고쳐서 다시 올려라. 특히 1번(P2 우회)은 변명의 여지가 없다.
