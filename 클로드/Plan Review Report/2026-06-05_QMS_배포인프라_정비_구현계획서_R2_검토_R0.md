## 4. 세부 분석

### [D-잔여] clone-db 소스 크리덴셜이 정의되지 않음 (🟠 중간 — 보완 필요)

W6 `clone-db.js`는 **Supabase JS 클라이언트**(`createClient(HTTP URL, key)`)로 메인을 읽는 구조다. 그런데 메인을 읽으려면 **메인의 HTTP URL + 메인 service-role 키**가 필요한데, `.env.local`에는 이게 없다. 현재 정의된 변수는:

```
VITE_SUPABASE_URL=<테스트 HTTP URL>
VITE_SUPABASE_ANON_KEY=<테스트 anon>
PROD_DB_URL=<postgres://...qrmy... 접속문자열>   ← W1 대상, 그러나 PG 문자열(JS 클라이언트가 안 씀)
STAGING_DB_URL=<postgres://...srza... 접속문자열>
SUPABASE_SERVICE_ROLE_KEY=<어느 프로젝트 키인지 불명>
```

문제는 **W1이 확정하는 건 `PROD_DB_URL`(postgres 접속문자열)인데, clone-db는 그걸 안 쓰고 HTTP URL+키를 쓴다**는 점이다. 즉 W1을 끝내도 clone-db의 소스 크리덴셜은 여전히 공백이다. 또 `SUPABASE_SERVICE_ROLE_KEY`가 테스트 것인지 메인 것인지 불명이라, 이대로면 clone-db가 메인을 못 읽거나 엉뚱한 DB를 읽을 수 있다.

**보완 요구**: clone-db의 ① 메인 HTTP URL ② 메인 service-role 키를 각각 **어느 환경변수에서 가져올지** 명시할 것. 해당 변수가 없으면 신설을 명시하고, `SUPABASE_SERVICE_ROLE_KEY`가 어느 프로젝트 키인지 규명할 것. (W1과 묶어 처리하면 깔끔함 — W1이 "메인 DB의 정식 좌표"를 PG 문자열뿐 아니라 HTTP URL+키까지 확정하도록 범위 확장 권고.)

### [I] Proxy 방어는 JS 레벨 한계 (🔵 낮음 — 심화 권고)

`createReadOnlyClient`의 `from()` 오버라이드는 좋은 1차 방어지만, `.rpc()`·다른 경로 호출은 못 막는다. 메인에 service-role 키를 쥐여주는 한 쓰기 권한 자체는 살아 있다. 가능하면 **메인 측에 read-only Postgres role**을 별도로 두고 그 키로만 접속하는 게 근본 방어다. 현 구조도 단일목적 스크립트엔 수용 가능하나, 잔여 위험으로 기록.

### [J] migrate "대체됨" 표현 부정확 (🔵 낮음)

R2 표는 `migrate-weekly-reports.js`를 clone-db로 "대체됨"이라 적었으나, 둘은 목적이 다르다(1회성 v1→v2 이관 vs 정기 메인→테스트 동기화). 기능 대체가 아니라 **임무 종료 + 메인 쓰기 위험**이 폐기 사유다. 표현만 바로잡으면 됨(처분 결정 자체는 타당).

### [F] .gemini 타임스탬프 독립검증 불가 (🔵 낮음 — 단서)

KI 비활성 증빙으로 첨부된 타임스탬프(`staging_status.md` 2026-04-16 등)는 `C:\Users\mjjeon\.gemini\` 경로로 **연결 폴더 밖**이라 코워크가 독립 확인하지 못했다. 플랜 제시값 자체는 형식상 합당하나, 최종 확인은 차장님이 해당 경로에서 직접 대조 권고.

**문구·심화(낮음)**
