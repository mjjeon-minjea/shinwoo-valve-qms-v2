## 3. 세부 분석

### [A] Task 1 — 코드가 실제 `server.js`와 불일치 (심각도: 🔴 높음)

플랜이 제시한 코드는 두 가지가 동시에 틀렸다.

**문제 1) `app`은 존재하지 않는 변수다.**
실제 `server.js`는 `const server = jsonServer.create();`로 인스턴스명이 **`server`**다. 플랜의 `app.use(...)`를 그대로 붙이면 `ReferenceError: app is not defined`로 **서버가 기동되지 않는다.**

**문제 2) 별도 미들웨어를 "앞에 추가"하는 접근 자체가 동작하지 않는다.**
플랜은 새 `app.use('/dev_notes', ...)` 블록을 추가하는 형태인데, 실제 검증 로직은 이미 48~67행에 독립 미들웨어로 존재한다.

```javascript
// 실제 server.js 48~67행 (현행)
server.use((req, res, next) => {
    if ((req.method === 'POST' || req.method === 'PATCH' || req.method === 'PUT') &&
        (req.path === '/dev_notes' || req.path.startsWith('/dev_notes/'))) {
        if (req.body.content !== undefined) {
            const content = req.body.content || '';
            const requiredKeywords = ['원인', '대책', '결과', '물리적 증빙'];
            const missing = requiredKeywords.filter(kw => !content.includes(kw));
            if (missing.length > 0) {
                return res.status(400).json({ error: `[시스템 락] DNAS 포맷 위반...` });
            }
        }
    }
    next();
});
```

Express에서 `return next()`는 "검증을 건너뛴다"가 아니라 "**다음 미들웨어로 넘어간다**"는 뜻이다. 즉 새 예외 미들웨어를 위에 추가해도, `next()`가 호출되면 곧바로 **기존 48~67행 검증기로 진입**해 반려 요청이 다시 400으로 막힌다. 별도 블록을 추가하는 방식으로는 절대 우회 목적을 달성할 수 없다.

**권고(반드시 이대로):** 새 미들웨어를 만들지 말고 **기존 48~67행 블록을 직접 수정**하여, content 검사 *이전에* 반려 예외를 끼워 넣어야 한다.

```javascript
server.use((req, res, next) => {
    if ((req.method === 'POST' || req.method === 'PATCH' || req.method === 'PUT') &&
        (req.path === '/dev_notes' || req.path.startsWith('/dev_notes/'))) {

        // 💡 반려(rejected)는 품질 키워드 검증 제외 — content 검사보다 먼저
        if (req.body && req.body.status === 'rejected') {
            return next();
        }

        if (req.body.content !== undefined) {
            // ...기존 키워드 검증 로직 그대로 유지...
        }
    }
    next();
});
```

이 수정이 잘못되면 Task 2(꼼수 텍스트 제거)도 함께 실패한다. 꼼수 텍스트를 빼면 본문에 키워드가 없어지는데, 미들웨어 예외가 동작하지 않으면 반려가 400으로 거부되기 때문이다. **Task 1과 Task 2는 운명 공동체다.**

### [B] Task 3 — free-text 날짜 입력의 포맷·정렬 위험 (심각도: ⚠️ 중간)

플랜은 `<input type="text" placeholder="예: 2026-04-13 14:00">`로 날짜를 자유 입력받는다. 그러나 `PostApproval.jsx` 15행의 정렬 함수가 `new Date(b.created_at || b.date)`로 날짜를 파싱한다. 사용자가 `2026.4.13`, `4/13`, `오늘` 등 임의 문자열을 입력하면 `Invalid Date`가 되어 **정렬이 깨지고 게시판 노출 순서가 망가진다.** 또한 기존 db.json 날짜는 `2026-01-29` 형식인데 free-text는 이를 보장하지 못해 포맷 혼재가 재발한다(이번에 고치려던 문제를 다시 만든다).

**권고:** `type="date"` 또는 `type="datetime-local"`을 사용하거나, 최소한 저장 직전 `YYYY-MM-DD` 정규화 + 유효성 검사를 추가할 것.

### [C] Task 3 — `customDate` 인풋 배치 위치 불명확 (심각도: ⚠️ 중간)

플랜은 "승인 모달 또는 승인 처리 행에 인풋 탑재"라고 했으나, 현재 `PostApproval.jsx`에서 **승인은 모달이 아니라 행(Row)의 직접 버튼**(`handleApprove(item)`, 330행)으로 처리된다. `selectedNote` 모달(365행~)은 닫기 버튼만 있고 승인 기능이 없다. 단일 `customDate` State를 화면 상단에 두면 "어느 행을 승인할 때 적용되는지" 불명확하고, 여러 건 승인 시 직전 입력값이 엉뚱한 노트에 적용될 수 있다.

**권고:** 승인 흐름을 모달 기반으로 옮겨 모달 안에 날짜 입력 + 승인 버튼을 함께 두거나, 행별 인풋으로 명시할 것. 현재 설계로는 UX 버그가 발생한다.

### [D] Task 5 — 전체 복사가 `node_modules` 포함 (심각도: ⚠️ 중간)

`Copy-Item -Path "...\*" -Recurse -Force`는 `node_modules`(수만 개 파일)까지 복사한다. Windows의 260자 경로 길이 제한에 걸려 복사가 중간에 실패하거나, 심볼릭 링크/캐시가 깨진 채 복사될 수 있다. 또한 `.env.local`(VITE_ 환경변수) 누락 시 빌드가 다른 이유로 실패해 **"한글 경로 탓"이라는 잘못된 결론**을 내릴 위험이 있다 — 이는 검증 목적 자체를 훼손한다.

**권고:** 소스만 복사(`node_modules` 제외)한 뒤 임시 경로에서 `npm install`을 새로 수행하고, `.env.local`을 명시적으로 복사할 것. 그래야 변수 = 경로 한글 여부만 남아 깨끗한 대조 실험이 된다.

### [E] P4(is_synced 구조 위험) 누락 — 이월 미표기 (심각도: 🔵 낮음, 이월)

직전 R0 검토에서 "보완 권장(중간)"으로 분류한 P4(Supabase에 배포상태 컬럼 부재 → 로컬 db.json 초기화 시 복구 불가)가 본 플랜에 포함되지 않았다. 의도적 후순위라면 문제없으나, 플랜 Goal에 "클라우드 연동 상의 DNAS 통제 무결성 강화"가 적혀 있는데 정작 클라우드 관련 Task가 0건이라 범위와 목표가 불일치한다.

**권고:** P4를 차기 리비전 후속 과제로 명시 이월하거나, Goal 문구에서 클라우드 항목을 빼서 범위를 일치시킬 것.

## 5. 📌 안티그래비티에게 — 조목조목 지적 댓글

> 실행 담당 안티그래비티가 R1 재작성 시 반드시 반영할 것.

**1. Task 1 — 너 이 코드 실제 `server.js` 열어보고 쓴 거 맞나?**
파일에는 `const server = ...`라고 똑똑히 적혀 있는데 플랜엔 `app.use`라고 썼더라. 이거 그대로 붙이면 `app is not defined`로 **서버가 안 켜진다.** 차장님이 실행 버튼 누르는 순간 죽는다는 얘기다. 추상적으로 "예외 미들웨어 추가" 떠올리지 말고, 고쳐야 할 48~67행을 직접 펼쳐놓고 그 안에 넣어라.

**2. Task 1 — 더 심각한 거. 미들웨어를 "앞에 하나 더" 추가하는 발상 자체가 틀렸다.**
Express에서 `return next()`는 검증을 건너뛰는 게 아니라 **다음 미들웨어로 넘기는 것**이다. 네 예외 블록이 `next()` 부르면 그 다음에 있는 기존 검증기로 그대로 들어가서 반려 요청이 또 400 맞는다. 즉 `app` 오타를 고쳐도 동작 안 한다. 별도 블록 만들지 말고 **기존 검증 블록 안에, content 검사보다 먼저** `if (status === 'rejected') return next();`를 끼워 넣어라. 이걸 틀리면 Task 2도 같이 죽는다 — 꼼수 텍스트 빼는 순간 키워드가 사라지니까.

**3. Task 3 — 날짜를 왜 `type="text"`로 받나?**
사용자가 `2026.4.13`이나 `4/13` 같은 거 치면 `new Date()`가 못 알아먹고 정렬(15행)이 박살난다. 그러면 이번에 고치겠다던 "날짜 포맷 왜곡"을 네 손으로 다시 만드는 거다. `type="date"` 써라. 그리고 승인은 모달이 아니라 행 버튼으로 처리되는데 인풋을 어디 둘 건지도 안 정했다 — 지금 설계로는 어느 노트에 날짜가 먹히는지 아무도 모른다.

**4. Task 5 — `node_modules`까지 통째로 복사하면 실험이 오염된다.**
Windows 경로 길이 제한에 걸려 복사가 깨지거나 `.env.local`이 빠지면, 빌드는 "한글 경로"가 아니라 엉뚱한 이유로 실패한다. 그걸 보고 "거봐 한글 경로 맞네"라고 결론 내면 그게 바로 오진이다. 소스만 복사하고 임시 폴더에서 `npm install` 새로 하고 `.env.local` 챙겨라. 변수는 경로 한글 여부 하나만 남겨야 깨끗한 실험이다.

**5. 범위 한마디 — Goal엔 "클라우드 무결성 강화"라고 써놓고 클라우드 Task는 0건이다.**
P4(is_synced 클라우드 미기록) 안 할 거면 "후속 이월"이라고 명시하고, Goal 문구에서 클라우드를 빼라. 안 할 일을 목표에 적어두면 그것도 형식주의다.

> **결론:** 방향은 잘 잡았다. 우선순위도 지난번 지적대로 고쳤더라. 그런데 Task 1은 실행하는 순간 서버가 죽거나 반려가 안 먹는다 — **이대로는 승인 못 해준다.** 위 1·2번 고쳐서 R1으로 다시 가져와라. 코드 쓸 땐 머릿속 말고 실제 파일을 열어놓고 써라.
