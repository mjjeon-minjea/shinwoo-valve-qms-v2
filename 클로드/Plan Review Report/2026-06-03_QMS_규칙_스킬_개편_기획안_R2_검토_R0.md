## 1. 검토 개요

본 기획안은 규칙(Rules)과 스킬(Skill)의 의존 구조를 분리(Decoupling)하고, 물리 제어 로직을 `revision-archiver` 스킬로 이관하여 Plan·Task·Walkthrough·Report·images "5대 도메인"을 단일 스킬로 통합 제어하는 것을 골자로 합니다. 방향성(규칙은 호출 조건만, 스킬은 기계 로직)은 타당합니다. 다만 **실제 워크스페이스 파일과 대조했을 때 AS-IS 인용 누락, 안전장치 삭제, 자동화 메커니즘의 실현 불가능성, 도메인 통합의 자기모순** 등 구현 전 반드시 잡아야 할 결함이 확인되었습니다.

## 3. 세부 분석

### [A] `plan-self-review` 필수 위임 항목 삭제 (🔴 높음)

**어디가:** §1 규칙 정비 → `[MODIFY] .agent/rules/02_dnas_process.md`의 "## 4. 스킬 실행 안내 및 자동 위임" TO-BE 블록.

**어떻게 문제인가:** 기획안의 AS-IS 인용에는 항목 4가 `revision-archiver`, `release-sync` **2개만** 적혀 있습니다. 그러나 실제 현행 파일에는 세 번째 줄이 더 있습니다:

```markdown
- **기획안 제출 전 자가검토 (필수)** ➔ `.agent/skills/plan-self-review/` — Plan 작성 완료 후
  차장님 송출 직전 반드시 실행, 높음 등급 이슈 0건 확인 후에만 제출 허용
```

기획안의 TO-BE는 이 줄을 통째로 빠뜨린 채 항목 4를 새로 써서 덮습니다. 그대로 적용하면 **"기획안 제출 전 필수 자가검토" 게이트가 규칙에서 삭제**됩니다. 이 게이트는 지금 검토 중인 결함들을 사전에 거르는 안전장치인데, 개편안 자체가 그 안전장치를 제거하는 자기모순입니다.

**대책방안:** TO-BE 블록에 `plan-self-review` 위임 줄을 **반드시 보존**합니다. 5대 도메인 문장을 추가하더라도 기존 3개 위임(revision-archiver / release-sync / plan-self-review)은 1:1로 유지하고, 변경 전 현행 파일을 diff로 재인용하여 누락이 없는지 교차검증해야 합니다.

### [B] 이미지 "즉시 강제 자동 복사" — 스킬로는 자동 발동 불가 (🔴 높음)

**어디가:** §1 `GEMINI.md [조항 6]` 신설 + §2 `revision-archiver` TO-BE의 images 항목("즉시 강제 자동 복사 실행").

**어떻게 문제인가:** 스킬(SKILL.md)은 **에이전트가 판단해 호출할 때만** 동작하는 위임 메커니즘이지, 도구 실행에 자동으로 끼어드는 후킹이 아닙니다. 따라서 `generate_image` 호출 "즉시 강제 자동"이라는 요구는 스킬 위임만으로는 보장되지 않습니다. 게다가 생성 이미지가 처음 어느 임시 경로에 떨어지는지(소스 경로)가 명세에 없어, 스킬이 무엇을 복사해야 하는지조차 정의되지 않았습니다.

**대책방안:** 진짜 "자동·강제"가 목표라면 위임 대상을 스킬이 아니라 **`.agent/hooks/`(워크스페이스에 이미 존재)** 의 후처리 훅으로 잡아야 합니다. 또는 "자동"을 포기하고 "이미지 생성 후 에이전트가 복사 스킬을 호출한다"로 규칙 문구를 현실에 맞게 약화합니다. 어느 쪽이든 ① 생성 이미지의 소스 경로, ② 복사 트리거(훅 이벤트), ③ 중복 파일명 처리 규칙을 명세에 추가해야 합니다.

### [C] 스킬 frontmatter 미갱신으로 도메인 확장이 발동 안 됨 (🟠 중간)

**어디가:** §2 `revision-archiver/SKILL.md` TO-BE.

**어떻게 문제인가:** 본문 프로토콜만 "5대 도메인 통합"으로 바꿨을 뿐, 현행 SKILL.md 상단 frontmatter의 `description`과 `triggers`는 여전히 `plan / walkthrough / 리비전`만 나열합니다(현행 확인 완료). images·task·report라는 트리거 키워드가 없으면 해당 상황에서 스킬이 선택·발동되지 않아, 규칙이 위임해도 스킬이 받지 못하는 **연동 단절**이 생깁니다.

**대책방안:** TO-BE에 frontmatter 수정도 포함시켜 `triggers`에 `task, report, images, 이미지` 등을 추가하고 `description`을 5대 도메인 통합 제어로 갱신합니다. 본문만 고치지 말고 frontmatter–본문을 한 세트로 변경 명세화합니다.

### [D] 리비전 로직 + 이미지 복사 로직 통합 → "혼재" 재발 (🟠 중간)

**어디가:** §1 아키텍처 TO-BE 다이어그램 + §2 스킬 TO-BE(이미지를 revision-archiver에 포함).

**어떻게 문제인가:** Plan·Task·Walkthrough·Report는 `_R(\d+)` 리비전을 매기는 **텍스트 문서**지만, images는 리비전 대상이 아닌 **바이너리 자산**으로 동작(즉시 복사)이 전혀 다릅니다. 성격이 다른 두 책임을 한 스킬에 묶으면, 이 기획안이 AS-IS에서 비판한 바로 그 "로직 혼재"를 TO-BE에서 다시 만드는 셈입니다.

**대책방안:** 이미지 복사는 별도 스킬(예: `image-archiver`) 또는 위 [B]의 훅으로 분리합니다. `revision-archiver`는 텍스트 4종 문서의 리비전 제어에만 집중시켜 단일 책임을 유지합니다.

### [E] "동일 주제명 감지" 판정 기준 부재 (🟠 중간)

**어디가:** §2 스킬 TO-BE "2. 명명 및 리비전 보증 공식" + §3 검증계획.

**어떻게 문제인가:** "스캔 결과 동일 주제명이 감지될 시 R을 +1" 한다고만 하고, 무엇을 "동일 주제"로 볼지 정규화 규칙이 없습니다. 정규식 `_R(\d+)`는 리비전 숫자만 뽑을 뿐 주제 동일성은 판단하지 못합니다. 실제로 이 기획안 주제명 `QMS_규칙_스킬_개편_기획안`과 스킬 예시의 `QMS_규칙_개편_기획안`은 한 단어("스킬") 차이로 **서로 다른 주제로 인식**되어, 같은 계열인데도 리비전이 이어지지 않고 R0로 새로 떨어집니다.

**대책방안:** 주제 정규화 키를 정의합니다(예: 날짜·`_R\d+` 제거 후 공백·언더스코어 정규화한 문자열을 비교 키로 사용, 대소문자·전각 공백 무시). 부분 일치 시 후보를 사용자에게 확인하는 분기도 명시합니다.

### [F] 리비전 시작값 불일치 (🔵 낮음)

**어디가:** 스킬 description("R0 → R1 → R2"), TO-BE 예시(`_R1.md`), 검증계획("R0 ➔ R1"), 그리고 본 기획안 파일명(`_R2`).

**어떻게 문제인가:** 최초 생성본이 R0인지 R1인지가 문서마다 다릅니다. 시작 기준이 불명확하면 자동 증가 로직의 첫 분기에서 오프바이원이 발생합니다.

**대책방안:** "신규 생성 = R0, 이후 +1" 또는 "R1 시작" 중 하나로 못 박아 스킬·검증계획·예시 표기를 통일합니다.
