## 1. 검토 개요

R2 검토에서 유일하게 남았던 중간 결함 **[D-잔여] clone-db 소스 크리덴셜 미정의**가 R3에서 해소됐다. W1 범위를 "메인의 정식 좌표(PG 문자열 + HTTP URL + service-role 키)"까지 확장하고, clone-db가 참조할 `PROD_SUPABASE_URL`·`PROD_SERVICE_ROLE_KEY`를 신설하도록 명시했으며, 기존 `SUPABASE_SERVICE_ROLE_KEY`가 테스트 DB 소속이라 메인 접근에 못 쓴다는 점을 JWT 디코드로 규명했다. **이 핵심 주장은 코워크 실측과 정확히 일치한다.** R0의 치명 3건부터 R2의 중간 1건까지 전부 정상 처리됐고, 새 결함은 발견되지 않았다. **승인 가능**하다.

## 3. 이번 R3 실측 검증

### [D-잔여] 해소 확인 — JWT `ref` 디코드 (✅ 일치)

R3는 `.env.local`의 service-role 키가 테스트 DB 소속이라 메인 접근에 쓸 수 없다고 주장했다. 코워크가 `.env.local`의 JWT 페이로드를 직접 디코드해 대조한 결과:

| 환경변수 | JWT `ref` | `role` | 판정 |
|---------|-----------|--------|------|
| `VITE_SUPABASE_ANON_KEY` | `srzaanvojyhwzugoaimk` | `anon` | 테스트 ✅ |
| `SUPABASE_SERVICE_ROLE_KEY` | `srzaanvojyhwzugoaimk` | `service_role` | **테스트 ✅ (메인 아님)** |

→ R3 주장과 **정확히 일치**. 따라서 clone-db가 메인을 읽으려면 별도의 `PROD_SUPABASE_URL`·`PROD_SERVICE_ROLE_KEY`가 반드시 필요하다는 R3의 설계가 타당하다. clone-db에 두 변수 미설정 시 `process.exit(1)`로 종료하는 가드까지 넣어, **W1 승인 전 실행이 물리적으로 불가**하도록 한 것도 적절하다.

### [K] 코스메틱 — Task4 삭제 사유표의 "기능 대체" 잔존 (🔵 매우 낮음 · 선택)

[J] 정정이 Task5 NOTE·관계표에는 반영됐으나, **Task4 삭제 사유표(245행)**에는 여전히 `migrate-weekly-reports.js | 신규 clone-db.js로 기능 대체`라고 남아 있다. Task5의 정정문("기능 대체 아님, 임무 종료+위험 폐기")과 한 문서 안에서 미세하게 어긋난다. 동작·안전엔 무관한 표현 일관성 문제이므로, 다음 수정 때 한 줄만 맞추면 된다(승인을 막을 사유 아님).
