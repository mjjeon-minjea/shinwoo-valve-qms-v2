## 1. 검토 개요

팩트체크에서 인정된 3건([E] clone-db 테이블 불일치 / [D] 미승인 DDL / [F] 비번 로테이션)에 대한 후속 플랜이다. 핵심은 `TARGET_TABLES`를 실측 스키마 기준 10종으로 교정하고, DDL 승인 게이트·비번 로테이션은 절차로 명문화하는 것. 플랜의 구조·롤백·DNAS 준수·잠긴 표면 보호는 양호하나, **승인 근거가 되는 "실측 스키마" 자체를 코워크 마운트에서 검증할 수 없고**, 이전 R0가 지적한 두 테이블이 설명 없이 사라진 점, [F]가 "즉시"에서 "리마인드"로 격하된 점이 승인 전 해소되어야 한다.

## 3. 세부 분석

### 검증된 사실 (실디스크 대조 ✅)
- 현행 `TARGET_TABLES` = 9종(`users, weekly_reports, inspections, item_master, process_inspections, calendar_events, inquiries, resources, suggestions`) — 플랜의 "현행 코드" 컬럼과 **정확히 일치**.
- 플랜 변경 = `+dev_notes, +notices, +settings` / `−calendar_events, −resources` → 10종. 내부 논리 일관됨.
- `GEMINI.md §5-2`에 "DB 스키마 직접 수정(DDL)"이 이미 인간통제 영역으로 명시됨 → **Task 2는 기존 규칙 재확인(비파괴)**. 무해하나 신규 통제는 아님.
- 잠긴 표면 4종은 플랜이 건드리지 않음(§5-1 부합) ✅.

### [A] 실측 스키마 교차검증 불가 (높음)
플랜 전체가 "2026-06-05 15:19 실측" 표에 의존한다. 그러나 이 표의 핵심 주장 — *정확히 이 10종이 메인·테스트 양쪽에 존재하고 `calendar_events`·`resources`는 테스트 DB에 없다* — 는 라이브 DB 쿼리로만 확인 가능하며 코워크 마운트로는 불가. **승인은 차장님 실터미널 라이브 쿼리 확인을 전제로만 유효**하다.

### [B] 사라진 두 테이블 (높음)
이전 검증 R0는 `dev_notes·notices·receiving_inspections·system_settings` 누락을 지적했다. 본 플랜은 앞 둘은 추가했으나 **`receiving_inspections`·`system_settings`는 표에 행조차 없다** (`⬜ 제외` 표기도 없음). 두 테이블이 실제로 양 DB에 없어 제외된 것인지, 누락된 것인지 플랜으로는 감사 불가. 만약 실재하고 앱이 쓴다면 [E]를 방향만 바꿔 재발시키는 셈. → **표에 두 테이블 행을 명시하고 존재 여부·제외 사유를 기재**해야 함.

### [C] `settings` 참조 0건 (중간)
src 전체에서 `settings` 직접 참조 0건(`dev_notes` 2, `notices` 1, `resources` 1). 앱이 쿼리하지 않는 테이블을 복제 대상에 넣는 것은 가능하나(서버/api 사용 등), 이전 R0의 지적이 `system_settings`였던 점을 감안하면 **`settings` vs `system_settings` 명칭 혼동 가능성**. 라이브 확인 시 정확한 테이블명 확정 필요.

### [D] [F] 격하 + 순서 미정의 (높음)
핸드오프는 [F]를 "즉시"로 분류했다 — 로테이션 전 메인 라이브 접근이 이미 발생, 노출 키 재발급이 목적. 플랜은 이를 코드변경 없는 "리마인드"로만 처리하고 완료 확인 단계가 없다. 또한 **Task 1 Step 3가 `PROD_SERVICE_ROLE_KEY`(노출 의심 키)로 clone-db를 재실행**하는데, [F] 로테이션과의 순서가 정의되지 않음. 로테이션이 먼저면 clone-db는 신규 키 필요. → [F]를 추적되는 즉시 과업으로 승격(로테이션 완료 + 구키 무효화 확인)하고 Task 1과의 키 순서를 명기.

### [E] "쓰기 0건" 미증명 (중간)
검증계획 #2의 기대값은 "메인 DB 쓰기 0건"이나, 이를 실제로 증명하는 단계가 없다. Step 4 대조 쿼리는 staging(`VITE_SUPABASE_URL`)만 조회. 메인 쓰기 차단은 코드의 Proxy 래퍼(insert/update/upsert/delete 물리 차단)가 담보하므로 실위험은 낮으나, 검증항목으로 두려면 메인측 행수 전후 비교 또는 "Proxy 차단 의존" 명기 필요. 또한 진짜 합격 기준인 *테이블별 메인↔staging 행수 정합*이 검증계획에 없음(빌드 성공은 테이블 목록 오류를 잡지 못함).

### [F] 행번호 드리프트 (낮음)
플랜의 `clone-db.js:59-69`, `:1-5`는 실디스크와 미세 어긋남(배열 실제 ~58행). 마운트 staleness 영향 가능 → 안티는 **행번호가 아닌 내용(`const TARGET_TABLES`)으로 위치 식별** 권고.
