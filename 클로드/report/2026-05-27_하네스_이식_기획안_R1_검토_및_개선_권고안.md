## 1. 총평

R0 대비 **획기적으로 개선**됐다. R0에서 지적했던 핵심 결함들—잠긴 표면 정의 부재, 롤백 경로 미비, 피드백 루프 미완결, 자동 삭제 위험, PowerShell 환경 미준수—이 R1에서 대부분 해소되었다. 문서의 완성도와 구조적 논리도 크게 향상됐다.

그러나 **코드베이스를 실제로 열어 확인하지 않고 작성된 가정들**이 여전히 잔존한다. 특히 3단계 피드백 루프의 기술적 기반이 현재 QMS v2의 실제 스택과 맞지 않는 부분이 있어, 그대로 구현에 들어가면 3단계 전체가 무효화될 수 있다. 이 점이 R1의 가장 큰 위험 요소다.

## 3. 잔존 문제점 분석

### 🔴 치명적 결함 — 3단계 전체 기술 기반 오류

R1의 가장 큰 문제는 **3단계 피드백 루프의 핵심 가정이 QMS v2 실제 코드베이스와 불일치**한다는 것이다. 직접 코드베이스를 확인한 결과는 다음과 같다.

**문제 1: QMS v2는 TypeScript 프로젝트가 아니다**

3단계 전체 전략이 Supabase CLI로 TypeScript 타입을 생성하고(`supabase.ts`), 프론트엔드의 TypeScript 제네릭과 대조하는 방식에 의존한다.

```powershell
# 기획안의 핵심 명령어
npx supabase gen types typescript --local > src/types/supabase.ts
```

그러나 실제 QMS v2를 확인하면:
- `src/` 전체에 `.ts` / `.tsx` 파일이 **단 1개도 없다** (전부 `.jsx`)
- `src/types/` 디렉토리 **자체가 존재하지 않는다**
- `package.json`에 TypeScript 관련 의존성이 **없다**

TypeScript 타입을 생성해도 비교할 프론트엔드 타입 파라미터가 존재하지 않는다. 3단계 전략의 전제가 무너진다.

**문제 2: `fetchJson<T>` 제네릭은 실재하지 않는다**

기획안은 "프론트엔드 React Hook(`fetchJson<T>`)의 타입 파라미터"를 스캔한다고 기술했으나, 실제 `src/lib/api.js`의 API 레이어는 `api.fetch()`라는 순수 JavaScript 함수이며 TypeScript 제네릭이 없다.

**문제 3: `--local` 플래그는 로컬 Supabase 인스턴스를 요구한다**

`npx supabase gen types typescript --local`은 Docker로 띄운 **로컬 Supabase 인스턴스**에서 스키마를 읽는 명령어다. QMS v2는 클라우드 호스팅 Supabase를 사용하므로, 이 명령어는 실행 즉시 실패한다. 원격 프로젝트라면 아래와 같이 사용해야 한다:

```powershell
# 올바른 명령어 (원격 Supabase 프로젝트)
npx supabase gen types typescript --project-id <PROJECT_ID> > src/types/supabase.ts
```

**3단계 대안 제안:**

TypeScript 없이 QMS v2에 맞는 현실적 피드백 루프는 두 가지다.

- **방안 A (경량):** `src/lib/api.js`의 테이블명 목록과 Supabase 대시보드 REST API 응답의 컬럼명을 대조하는 JavaScript 기반 스크립트 작성. 별도 CLI 도구 없이 `node-fetch`로 스키마 정보를 가져올 수 있다.
- **방안 B (근본 해결):** 3단계에 착수하기 전에 QMS v2에 TypeScript를 점진 도입(`allowJs: true` 모드)하고, 이후 타입 기반 검증을 구축. 단, 이는 2단계와 3단계 사이에 별도의 "TS 도입 단계"가 필요하다.

---

### 🟠 중요 결함 — 2단계 Zone 정의가 실제 폴더 구조와 불일치

기획안의 Zone 경계 정의:
```
ui (src/components/, src/pages/) → hooks/ → lib/ (src/lib/)
```

실제 QMS v2 폴더 구조를 확인한 결과:
- `src/pages/` 디렉토리 **존재하지 않음** (모든 화면 컴포넌트가 `src/components/`에 존재)
- `src/hooks/` 디렉토리 **존재하지 않음**

Zone 정의가 없는 폴더를 기준으로 만들어지면 `eslint-plugin-boundaries`가 해당 zone을 인식하지 못해 제약이 사실상 무력화된다. Zone은 **실재하는 폴더 구조를 기준**으로 재정의해야 한다.

**수정 제안:**
```
zone: components (src/components/) → lib (src/lib/)
(hooks 폴더가 없으므로 중간 레이어 생략, 추후 신설 시 추가)
```

---

### 🟠 중요 결함 — `CLAUDE.md`가 변경 영향 범위에만 있고 실체가 없다

1절 "변경 영향 범위"에 `CLAUDE.md [NEW]`가 명시되어 있으나, **5단계 어디에도 CLAUDE.md를 작성하는 단계가 없다.** 무엇을 담는 파일인지, 어느 단계에서 생성하는지, 어떤 형식인지가 전혀 정의되어 있지 않다. 이 상태에서 에이전트가 CLAUDE.md를 작성하면 형식과 내용이 임의로 결정된다.

---

### 🟡 경미한 결함 — `rm` 명령어는 PowerShell 문법이 아니다

롤백 시나리오 표의 5단계 항목:
> 감지된 오염 파일을 백업 폴더(`_workspace/backup/`)로 **수동 이동 처리**

본문에서는 "수동 이동"이라고 했지만, 5단계 구현 사양 3번 항목에 `rm 대행 명령`이라는 표현이 있다. `rm`은 Linux/Mac 명령어이며, PowerShell에서는 `Remove-Item`을 사용해야 한다. R0 검토에서 이미 지적된 사항이 완전히 해소되지 않았다.

**수정:**
```powershell
# Linux (사용 금지)
rm ./src/temp_test.js

# PowerShell (사용)
Remove-Item -Path ".\src\temp_test.js"
```

---

### 🟡 경미한 결함 — "Router v7(React-Query)" 혼용

4단계 지식 저장소에서 캐싱 전략을 설명하며 "Router v7(React-Query) 기반 캐싱 전략"이라고 기술했다. 그러나 **React Router v7과 React Query(TanStack Query)는 전혀 다른 라이브러리**다.

- React Router v7: URL 라우팅 + 서버 Data Loader 제공
- React Query: 서버 상태 캐싱 및 동기화 전용 라이브러리

또한 현재 `package.json`에는 TanStack Query / React Query가 **설치되어 있지 않다.** 실제로 사용하고 있는 캐싱 전략이 무엇인지를 확인한 후 정확한 기술명으로 ADR을 작성해야 한다.

---

### 🟡 경미한 결함 — husky/lint-staged 설정 절차 누락

기획안은 설치 명령어만 있고 실제 설정 절차가 없다. `husky`는 설치 후 반드시 초기화(`npx husky init`)와 훅 파일 생성이 필요하며, `lint-staged`는 `package.json`에 별도 설정 블록을 추가해야 한다. 이것이 없으면 설치만 되고 작동은 안 된다.

**누락된 설정 절차:**
```powershell
# husky 초기화
npx husky init

# .husky/pre-commit 파일 내용
npx lint-staged
```

```json
// package.json에 추가해야 할 lint-staged 설정
"lint-staged": {
  "src/**/*.{js,jsx}": ["eslint --fix", "git add"]
}
```

## 4. 종합 평가

| 항목 | R0 | R1 |
|---|---|---|
| 방향성 | ✅ 맞음 | ✅ 맞음 |
| 잠긴 표면 설계 | ❌ 없음 | ✅ 있음 |
| 롤백 경로 | ❌ 없음 | ✅ 있음 |
| 실물 코드베이스 검증 | ❌ 미확인 | 🔴 **일부 미확인** |
| 3단계 기술 기반 | ❌ 정적 파싱 오류 | 🔴 **TypeScript 가정 오류** |
| Zone 정의 정확성 | - | 🟠 실존 폴더 불일치 |
| PowerShell 준수 | ❌ 미준수 | 🟡 rm 잔존 |
| 전체 완성도 | 40점 | **73점** |

R1은 구조와 논리 면에서 R0보다 훨씬 성숙했다. 그러나 3단계의 TypeScript 가정 오류는 단순 수정 사항이 아니라 **3단계 전략 자체를 재설계해야 하는 수준**의 문제다. 이것이 해결되지 않으면 하네스의 핵심인 피드백 루프가 작동하지 않는다.

## 5. 안티그래비티를 위한 추가 조언

R1에서 반복된 실수의 공통 원인은 하나다.

> **"기획안을 작성하면서 실제 파일을 열어보지 않았다."**

3단계의 TypeScript 오류, Zone 정의의 pages/hooks 폴더 오류 모두 해당 경로를 한 번만 확인했으면 발견할 수 있었던 것들이다. R2를 작성하기 전에 아래 명령어를 먼저 실행하고 결과를 직접 눈으로 확인한 후 기획을 시작하기 바란다.

```powershell
# QMS v2 src 폴더 구조 전체 확인 (필수)
Get-ChildItem -Path ".\src" -Recurse -Directory

# TypeScript 파일 존재 여부 확인 (필수)
Get-ChildItem -Path ".\src" -Recurse -Include "*.ts","*.tsx"

# 현재 설치된 패키지 확인 (필수)
Get-Content .\package.json
```

이 세 가지 확인만 했어도 R1의 치명적 결함 두 개는 사전에 막을 수 있었다.
